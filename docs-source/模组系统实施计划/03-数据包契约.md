## 6. 数据包契约

### 6.1 清单文件

每个数据包必须包含 `manifest.json`：

```json
{
  "id": "example_mod",
  "name": {
    "key": "example_mod.package.name",
    "fallback": "示例内容包"
  },
  "version": "1.0.0",
  "gameVersion": "2.4.0",
  "engineApiVersion": "1",
  "contentSchemaVersion": "1",
  "defaultLocale": "zh-CN",
  "locales": {
    "zh-CN": "locales/zh-CN.json",
    "en-US": "locales/en-US.json"
  },
  "authors": [
    { "name": "示例作者", "role": "developer" }
  ],
  "license": "MIT",
  "homepage": "https://example.com/mod",
  "issues": "https://example.com/mod/issues",
  "dependencies": [
    { "id": "example_library", "version": "1.3.2" }
  ],
  "entrypoints": {
    "taoyuan:tag": ["data/tags.json"],
    "taoyuan:item": ["data/items.json"],
    "taoyuan:crop": ["data/crops.json"],
    "taoyuan:tree": ["data/trees.json"],
    "taoyuan:fish": ["data/fish.json"],
    "taoyuan:pondable_fish": ["data/pondable-fish.json"],
    "taoyuan:pond_breed": ["data/pond-breeds.json"],
    "taoyuan:forage": ["data/forage.json"],
    "taoyuan:animal": ["data/animals.json"],
    "taoyuan:monster": ["data/monsters.json"],
    "taoyuan:monster_pool": ["data/monster-pools.json"],
    "taoyuan:enchantment": ["data/enchantments.json"],
    "taoyuan:drop_table": ["data/drop-tables.json"],
    "taoyuan:recipe": ["data/recipes.json"],
    "taoyuan:shop": ["data/shops.json"],
    "taoyuan:shop_offer": ["data/shop-offers.json"]
  },
  "settings": "settings.schema.json",
  "attributions": "attributions.json",
  "assets": "assets"
}
```

`entrypoints` 的键必须是已经定义的 `RegistryTypeId`。普通数据包遇到未知注册表类型时应拒绝加载，不能根据文件内容临时创建注册表。

首版清单至少包含：

- 唯一数据包 ID。
- 可本地化的用户可见名称和确定版本。
- 制作和测试该模组时所针对的游戏版本。
- 所需引擎 API 版本和数据 Schema 版本。
- 必需依赖和版本范围。
- 各注册表的数据入口。
- 默认语言、语言包入口、作者、许可证和问题反馈地址。
- 可选配置 Schema 和资源署名清单。
- 可选资源目录。

`loadBefore`、`loadAfter` 和可选依赖可以后续增加。首版应优先使用显式依赖排序，不通过模糊优先级掩盖冲突。

### 6.2 注册表行为

注册表类型本身也使用命名空间 ID。官方首批注册表类型包括：

```text
taoyuan:tag
taoyuan:item
taoyuan:crop
taoyuan:tree
taoyuan:fish
taoyuan:pondable_fish
taoyuan:pond_breed
taoyuan:forage
taoyuan:animal
taoyuan:monster
taoyuan:monster_pool
taoyuan:enchantment
taoyuan:drop_table
taoyuan:recipe
taoyuan:shop
taoyuan:shop_offer
```

内容的完整身份是“注册表类型 ID + 内容 ID”，而不是只有内容 ID：

```ts
type RegistryTypeId = NamespacedId

interface RegistryEntryKey {
  registryId: RegistryTypeId
  contentId: ContentId
}
```

因此 `taoyuan:item + taoyuan:seed` 与 `taoyuan:tag + taoyuan:seed` 可以同时存在。字段类型和目标注册表决定一个 ID 表示物品、标签还是配方，不在内容 ID 中重复写入 `tag::`、`item::` 等类别前缀。

通用注册表至少提供：

```ts
interface Registry<T> {
  register(owner: PackageId, entry: T): void
  get(id: ContentId): T | undefined
  require(id: ContentId): T
  has(id: ContentId): boolean
  values(): readonly T[]
  freeze(): void
}
```

`RegistrySet` 应作为按 `RegistryTypeId` 查询的注册表目录，不应将 `items`、`recipes` 等写成不可扩展的固定属性：

```ts
interface RegistrySet {
  defineRegistry<T>(definition: RegistryDefinition<T>): void
  get<T>(registryId: RegistryTypeId): Registry<T>
  freezeDefinitions(): void
  freezeEntries(): void
}
```

第一阶段只有引擎可以调用 `defineRegistry()`，数据包只能向引擎已公开的注册表注册条目。注册阶段保留内容来源、文件路径和原始本地 ID，以便错误报告能定位到数据包。冻结后不得暴露可修改数组和对象引用。

#### 6.2.1 怪物池契约

`taoyuan:monster_pool` 与 `taoyuan:monster` 相互独立。公开 TypeBox 契约的等价形状为：

```ts
interface MonsterPoolEntry {
  monsterId: ContentId
  weight?: number
}

interface MonsterPoolDef {
  id: ContentId
  entries: readonly [MonsterPoolEntry, ...MonsterPoolEntry[]]
}
```

- `weight` 必须是正整数，省略时按 `1` 处理。
- 单池 `entries` 原始条目数不超过 250,000，单条 `weight` 不超过 250,000，全部条目的有效权重总和不超过 250,000。
- 任一资源边界超限时产生 `PKG-LIMIT-001`，诊断包含 `poolId`、`actual`、`limit` 和 `fieldPath`；不得截断、归一化、清零或忽略超限权重。
- `entries` 必须非空，其顺序是公开且可观察的行为；注册、快照和解析均不得排序。
- 池只描述候选怪物和权重，不包含楼层生成、特殊层、属性缩放、战斗公式或随机数生成器。
- 每个 `monsterId` 必须引用 `taoyuan:monster` 中存在的条目，否则产生 `REG-REFERENCE-001`。
- 当前主矿洞 BOSS 池必须恰好包含一个等权条目，不满足时产生 `SCHEMA-VALIDATE-001`。
- 游戏运行所需的官方池缺失时产生 `REG-REQUIRED-001` `fatal` 诊断，不得在运行时静默退化为空池。
- 外部怪物池 JSON 必须先作为 `unknown` 处理，再通过由 `MonsterPoolDefSchema` 生成的 JSON Schema 和 Ajv 校验。

阶段 5 冻结以下 15 个官方池 ID：

```text
taoyuan:pool/mine/zone/shallow
taoyuan:pool/mine/zone/frost
taoyuan:pool/mine/zone/lava
taoyuan:pool/mine/zone/crystal
taoyuan:pool/mine/zone/shadow
taoyuan:pool/mine/zone/abyss
taoyuan:pool/mine/boss/20
taoyuan:pool/mine/boss/40
taoyuan:pool/mine/boss/60
taoyuan:pool/mine/boss/80
taoyuan:pool/mine/boss/100
taoyuan:pool/mine/boss/120
taoyuan:pool/skull-cavern/base
taoyuan:pool/skull-cavern/depth-11
taoyuan:pool/skull-cavern/boss
```

`depth-11` 只保存 `floor > 10` 时追加的候选；深度阈值、每 25 层出现 BOSS 和骷髅矿穴缩放仍由框架决定。当前官方池全部省略 `weight`，因此固定随机值继续映射到与旧数组下标完全相同的怪物。

#### 6.2.2 树木契约

`taoyuan:tree` 独立于 `taoyuan:crop` 和 `taoyuan:item`，使用 TypeBox 判别联合表达果树与野树：

```ts
type TreeDef =
  | {
      id: ContentId
      kind: 'fruit'
      name: LocalizedTextRef
      seedItemId: ContentId
      growthDays: number
      saplingPrice: number
      fruitItemId: ContentId
      fruitName: LocalizedTextRef
      fruitSeason: Season
      fruitSellPrice: number
    }
  | {
      id: ContentId
      kind: 'wild'
      name: LocalizedTextRef
      seedItemId: ContentId
      growthDays: number
      tapProductItemId: ContentId
      tapProductName: LocalizedTextRef
      tapCycleDays: number
    }
```

- `growthDays` 和 `tapCycleDays` 必须是正整数；`saplingPrice` 和 `fruitSellPrice` 必须是非负整数。
- 两个联合分支均使用 `additionalProperties: false`，果树和野树字段不得混合。
- `seedItemId`、`fruitItemId` 和 `tapProductItemId` 必须引用现有 `taoyuan:item`，缺失时产生 `REG-REFERENCE-001`。
- 外部树木 JSON 在校验前保持为 `unknown`，通过 `TreeDefSchema` / `tree.schema.json` 后才允许注册。
- 阶段 6 首切片冻结 11 个本地 ID：`peach_tree`、`lychee_tree`、`mandarin_tree`、`plum_tree`、`apricot_tree`、`pomegranate_tree`、`persimmon_tree`、`hawthorn_tree`、`pine`、`camphor`、`mulberry`。
- 注册表 ID 使用 `taoyuan:*`，兼容对象和 `PlantedFruitTree.type` / `PlantedWildTree.type` 继续使用上述本地 ID，不改变存档 JSON。
- `treeDefinitions.ts` 是这 11 种树木旧格式数值的唯一静态来源；`items.ts` 为派生树苗/果实物品、`staticAdapters.ts` 为官方投影、等价测试为回归基线而读取该叶子模块。普通 Store、composable 和 Vue 页面不得直接读取旧数组。

#### 6.2.3 采集物契约

`taoyuan:forage` 描述竹林采集物本身，不包含天气倍率、技能加成、工具消耗、动物遭遇或战斗流程。公开 TypeBox 契约的等价形状为：

```ts
interface ForageDef {
  id: ContentId
  itemId: ContentId
  name: LocalizedTextRef
  season: Season[]
  chance: number
  expReward: number
}
```

- `id` 是采集条目 ID；当前官方条目与产物 `itemId` 保持一致，但运行时背包和存档仍只保存本地物品 ID。
- `itemId` 必须引用现有 `taoyuan:item`，缺失时产生 `REG-REFERENCE-001`。
- `season` 必须非空且只使用四季枚举；`chance` 取值范围为 0 到 1；`expReward` 必须是非负整数。
- `WEATHER_FORAGE_MODIFIER`、`FOREST_ENCOUNTER_CHANCE`、动物/野兽遭遇随机、战败惩罚和战斗算法继续属于框架规则，不能随本采集物切片伪装成内容数据。
- 外部采集物 JSON 在校验前保持为 `unknown`，通过 `ForageDefSchema` / `forage.schema.json` 后才允许注册。

#### 6.2.4 动物定义契约

`taoyuan:animal` 描述可饲养动物的物种定义，不包含畜舍建造/升级、饲料效果、孵化器进度、放牧、疾病、死亡、宠物拾取或每日结算算法。公开 TypeBox 契约的等价形状为：

```ts
interface AnimalDef {
  id: ContentId
  name: LocalizedTextRef
  building: 'coop' | 'barn' | 'stable'
  cost: number
  productItemId?: ContentId
  productName?: LocalizedTextRef
  produceDays: number
  friendship: { min: number; max: number }
}
```

- `id` 是动物物种 ID；当前官方条目继续使用 `chicken`、`cow`、`horse` 等本地 ID 的 `taoyuan:*` 命名空间形式，运行时动物实例和旧存档仍保存本地 `type`。
- `building` 只表达物种归属建筑；建筑定义、容量升级、材料消耗和马厩建造仍由现有设施/框架规则处理。
- `productItemId` 仅对会产出的动物存在，必须引用现有 `taoyuan:item`，缺失时产生 `REG-REFERENCE-001`；马等无产出动物省略该字段且 `produceDays` 为 `0`。
- `cost`、`produceDays` 和 `friendship.min/max` 必须是非负整数；当前切片不改变购买价、产物、产出周期、好感范围、品质计算或出售退款。
- 外部动物 JSON 在校验前保持为 `unknown`，通过 `AnimalDefSchema` / `animal.schema.json` 后才允许注册。

#### 6.2.5 可养殖鱼定义契约

`taoyuan:pondable_fish` 描述可放入鱼塘养殖的鱼种定义，不包含鱼塘建造/升级、容量、水质衰减、疾病、死亡、育苗塘、繁衍塘或基因随机算法。公开 TypeBox 契约的等价形状为：

```ts
interface PondFishGenetics {
  weight: number
  growthRate: number
  diseaseRes: number
  qualityGene: number
  mutationRate: number
}

interface PondableFishDef {
  id: ContentId
  fishItemId: ContentId
  name: LocalizedTextRef
  maturityDays: number
  baseProductionRate: number
  productItemId: ContentId
  defaultGenetics: PondFishGenetics
}
```

- `id` 是可养殖鱼定义 ID；当前官方条目与 `fishItemId` 保持一致，运行时鱼塘实例和旧存档仍保存本地 `fishId`。
- `fishItemId` 必须引用现有 `taoyuan:fish`，`productItemId` 必须引用现有 `taoyuan:item`，缺失时产生 `REG-REFERENCE-001`。
- `maturityDays` 必须是正整数；`baseProductionRate` 取值范围为 0 到 1；`weight`、`growthRate`、`diseaseRes` 和 `qualityGene` 为 0 到 100 的整数；`mutationRate` 为 1 到 50 的整数。
- 当前切片只迁移 13 种 `PONDABLE_FISH` 物种定义；鱼塘容量、鱼苗上限、水质、喂食、疾病、产出概率加成、品质计算、育苗和自然繁衍仍由现有 Store/框架规则处理。
- 外部可养殖鱼 JSON 在校验前保持为 `unknown`，通过 `PondableFishDefSchema` / `pondable-fish.schema.json` 后才允许注册。

#### 6.2.6 鱼塘品种定义契约

`taoyuan:pond_breed` 描述鱼塘图鉴中的品种定义和亲本链路，不包含鱼塘建造/升级、容量、水质、疾病、死亡、育苗塘、繁衍塘、遗传随机、基因变异或品质计算算法。公开 TypeBox 契约的等价形状为：

```ts
interface PondBreedDef {
  id: ContentId
  name: LocalizedTextRef
  generation: 1 | 2 | 3 | 4 | 5
  baseFishId: ContentId
  parentBreedA: ContentId | null
  parentBreedB: ContentId | null
}
```

- `id` 是品种图鉴 ID；当前官方条目继续使用 `g1_001` 至 `g5_020` 的本地 ID 兼容形状，运行时鱼个体和旧存档仍保存本地 `breedId`。
- `baseFishId` 必须引用现有 `taoyuan:pondable_fish`，缺失时产生 `REG-REFERENCE-001`。
- `parentBreedA` 和 `parentBreedB` 对一代品种为 `null`；二代及以上品种引用同一 `taoyuan:pond_breed` 中存在的亲本，缺失时产生 `REG-REFERENCE-001`。
- 品种顺序、亲本配对和每代数量是玩家可观察的图鉴/繁育行为，注册表投影不得排序或重写。
- 当前切片只迁移 400 条 `POND_BREEDS` 品种定义和旧查询入口；育苗塘任务、自然繁衍进度、遗传随机调用、基因上下限和鱼塘存档结构仍由现有 Store/框架规则处理。
- 外部鱼塘品种 JSON 在校验前保持为 `unknown`，通过 `PondBreedDefSchema` / `pond-breed.schema.json` 后才允许注册。

### 6.3 冲突策略

默认策略如下：

1. 同一数据包在同一注册表内重复 ID：拒绝加载该数据包。
2. 不同数据包在同一注册表内使用同一完整 ID：拒绝后加载的数据包。
3. 未声明依赖却引用其他数据包内容：拒绝加载。
4. 官方内容 ID：第三方数据包默认不得覆盖。
5. 将来如支持覆盖，必须由清单显式声明，并在模组管理界面警告。

冲突检查以 `(registryId, contentId)` 为键；不同注册表可以拥有相同 `contentId`。注册表类型 ID 自身必须全局唯一，第三方插件不得定义 `taoyuan:*` 注册表。

### 6.4 引用解析

完整生命周期采用“先定义注册表类型，再注册内容条目”的顺序：

1. 引擎定义官方注册表类型。
2. 后续获得授权的事件插件定义私有注册表类型，机制插件定义自有公开注册表类型。
3. 冻结注册表类型目录，不再接受新的注册表定义。
4. 官方内容包和第三方数据包注册原始条目，此时不要求被引用项已经出现。
5. 所有数据包注册完成后，统一检查物品、标签、配方材料、掉落表、附魔和商店商品引用。
6. 冻结所有注册表内容，再创建 Store、读取存档并挂载 Vue。

这可以支持跨文件和跨依赖包引用，同时避免加载顺序偶然影响校验结果。

### 6.5 内容编译器

当前内置数据存在自动生成内容的逻辑。迁移后由内容编译器负责：

- 根据作物定义生成种子物品和收获物品。
- 根据烹饪配方生成成品物品。
- 根据加工配方生成加工品目录项。
- 建立按 ID 查询的 Map 和分类索引。
- 生成只读、稳定排序的运行时目录。

派生项必须有确定的 ID 规则，并检查是否与显式内容冲突。相同输入应产生相同输出，不得在编译阶段使用随机数。

### 6.6 标签化材料与配方选择器

物品标签用于表达烹饪和加工语义，例如“素食”“肉类”“蛋白质”“水果类”。标签不替代现有 `category`：分类继续服务于背包分组、出售和 UI，一个物品只有一个主要分类；标签是可扩展的多值集合，一个物品可以同时属于多个标签。

官方标签使用命名空间 ID，例如：

```text
taoyuan:vegetarian
taoyuan:meat
taoyuan:protein
taoyuan:fruit
```

第三方物品若希望被官方标签配方识别，应显式声明对应官方标签。数据包也可以注册自有标签，但引用其他数据包标签时必须声明依赖。标签定义、显示名称和引用关系由 `TagRegistry` 管理。

物品定义通过可选标签列表保持对现有内容的向后兼容：

```ts
interface ItemDef {
  id: ContentId
  tags?: ContentTagId[]
}
```

配方材料采用可辨识联合类型，保留现有固定物品材料，并增加标签和多标签择一材料：

```ts
type RecipeIngredient =
  | {
      type: 'item'
      itemId: ContentId
      quantity: number
    }
  | {
      type: 'tag'
      tagId: ContentTagId
      quantity: number
    }
  | {
      type: 'anyOfTags'
      tagIds: ContentTagId[]
      quantity: number
    }
```

旧配方中没有 `type`、只有 `itemId + quantity` 的材料，在内容标准化阶段按 `type: 'item'` 读取，避免为现有配方制造一次性破坏性迁移。

“包子”可以表达为：

```json
{
  "id": "taoyuan:steamed_bun",
  "ingredients": [
    {
      "type": "item",
      "itemId": "taoyuan:wheat_flour",
      "quantity": 1
    },
    {
      "type": "anyOfTags",
      "tagIds": ["taoyuan:vegetarian", "taoyuan:meat"],
      "quantity": 1
    }
  ]
}
```

标签配方不能分别统计每个材料槽后直接相加。同一物品可能同时拥有多个标签，例如豆腐可同时属于“素食”和“蛋白质”；同一份豆腐不能被两个材料槽重复占用。应在 `domain/` 中建立纯材料分配器，统一生成原子扣料计划：

```text
配方选择器 + 当前库存 + 制作数量
                ↓
        IngredientAllocationPlan
                ↓
具体扣除的物品、品质和数量
```

`canCook()`、最大可制作数量、品质预览和实际扣料必须复用同一规划入口。规划失败时不得扣除部分材料；批量制作必须一次生成完整计划。默认自动选料顺序应稳定，并优先使用低品质、低价值材料；烹饪弹窗应展示标签槽实际选择的物品，并允许玩家在执行前更换。成品品质继续取实际消耗材料中的最低品质。

#### 制品组成标签与堆叠

标签可以声明是否传播到制品，以及是否影响堆叠：

```ts
interface TagDef {
  id: ContentTagId
  propagateToCraftedOutput?: boolean
  stackPolicy?: 'merge' | 'separate'
}

interface CraftedItemStack {
  itemId: ContentId
  quantity: number
  quality: Quality
  compositionTags?: ContentTagId[]
}
```

配方仍产出一个基础物品定义，例如 `taoyuan:steamed_bun`，但每批产物根据实际消耗材料生成带 `compositionTags` 的运行时变体。只有 `propagateToCraftedOutput: true` 的材料标签会传播，避免把商店分类、内部索引等技术标签带入成品。

堆叠签名由以下字段组成：

```text
基础 itemId + 品质 + 所有 stackPolicy=separate 的组成标签（稳定排序）
```

`stackPolicy` 默认为 `merge`。两个制品的普通组成标签不同仍可堆叠；合并后 `compositionTags` 取去重并集，直接丢失每份制品的具体材料来源和逐份标签分布。拆分该堆物品时，拆出的各部分都继承合并后的标签并集，不尝试恢复已经丢失的来源。

`stackPolicy: separate` 的特殊标签必须参与堆叠签名。任一特殊标签集合不同，两个制品就不得合堆。例如 `taoyuan:human_meat` 必须定义为 `separate`，因此含“人肉”的包子、料理或加工品不能与不含该标签的同类制品堆叠。

任何会改变食用效果、售价、NPC 反应、心情、合法性或其他逐件玩法行为的组成标签，都必须使用 `separate`；`merge` 标签只能承受合并后来源丢失和标签并集语义。内容校验器应拒绝将行为敏感标签配置为 `merge`。旧存档没有 `compositionTags` 时按空数组读取。

#### 商店定义与商品 Offer

`ShopDef` 描述商店本身的显示名称、NPC、营业时间和天气/季节/星期条件；`ShopOfferDef` 描述某个商店内可售商品。阶段 3 的迁移先把官方六个既有商店和它们当前商品池投影进 `taoyuan:shop` 与 `taoyuan:shop_offer`，不改变购买动作、价格、解锁条件、音效、日志或数量上限。

`ShopOfferDef` 至少包含 `shopId`、`itemId` 和 `price`；可选字段用于保留现有 UI 语义：

- `name`、`description`：商品在当前商店中的显示覆盖文案，不替代物品定义本身。
- `groupId`、`groupName`：稳定显示分组，例如当季种子、武器、鱼饵、肥料、帽子和鞋子。
- `purchaseKind`：购买处理入口提示，当前限定为 `item`、`seed`、`weapon`、`hat` 或 `shoe`；实际扣钱、加物品、装备弹窗和日志流程仍由既有 Store/View 处理。
- `sortOrder`：同一商店内的稳定排序键；相同排序值再按内容 ID 排序。
- `availableSeasons`：只表达季节性商品可见性，不替代 `ShopDef` 的营业条件。

迁移期查询门面可以按 `shopId` 和当前季节返回 Offer 或分组后的 Offer，但不得让第三方数据包绕过公开 Schema 与语义引用校验。无效 `itemId` 应在注册表校验阶段报告，不得等玩家点击购买时才失败。

### 6.7 自定义注册表的开放边界

第一阶段的数据包只允许向现有注册表添加条目，不允许定义新的全局注册表。单独建立一个注册表不会自动产生玩法价值；游戏还必须知道如何校验、读取、执行、展示、保存以及在插件缺失时恢复其中的数据。

自定义数据按消费者能力分为三个等级：

| 需求 | 所属阶段 | 注册表权限 |
| --- | --- | --- |
| 向物品、标签、配方等现有类型添加条目 | 第一阶段：数据插件 | 只能注册条目 |
| 自定义数据只由插件自己的事件处理器读取 | 第二阶段：事件插件 | 可拥有受限的插件私有注册表 |
| 自定义数据对应新副本、设施、UI 或系统规则 | 第三阶段：机制插件 | 可申请定义公开注册表 |

例如，新增“魔法苹果”只需要注册到 `taoyuan:item`；事件插件内部使用的“法术定义”可以放入 `example_mod:spell` 私有注册表；若法术需要独立页面、法力值、战斗规则和跨插件引用，则属于机制插件。

未来开放自定义公开注册表时，应要求插件显式申请 `registry.define` 能力，并提供注册表所有者、Schema、版本和缺失插件策略：

```json
{
  "id": "example_mod",
  "capabilities": ["registry.define"],
  "registries": [
    {
      "id": "example_mod:spell",
      "schema": "schemas/spell.json",
      "schemaVersion": "1"
    }
  ]
}
```

数据插件不得通过清单声明、未知 `entrypoints` 键或内嵌 Schema 绕过该能力检查。自定义注册表必须先于内容条目冻结，且其跨插件引用、存档兼容、禁用行为和卸载恢复规则必须纳入机制插件验收。

### 6.8 官方注册表全量盘点

`TagRegistry`、`ItemRegistry`、`CropRegistry`、`MonsterRegistry`、`EnchantmentRegistry`、`DropTableRegistry`、`RecipeRegistry` 和 `ShopOfferRegistry` 只是首个迁移批次，不代表官方内容包的最终范围。阶段 0 必须逐文件盘点 `src/data/`，为所有静态内容确定目标注册表、派生编译器或保留在框架中的算法归属。

最低盘点范围包括：

| 内容组 | 当前内容示例 | 目标 |
| --- | --- | --- |
| 农业与自然资源 | 作物、鱼类、采集、果树、野树、动物、鱼塘、育种 | 建立对应定义注册表及引用校验 |
| 角色与叙事 | NPC、隐藏 NPC、好感事件、剧情任务、普通任务、提示、秘密纸条 | 建立角色、事件和任务注册表 |
| 成长与收藏 | 成就、技能相关内容、博物馆、行会、教程 | 建立进度与收藏内容注册表 |
| 生产与装备 | 加工机、加工配方、建筑、升级、武器、戒指、帽子、鞋、套装 | 建立生产、设施和装备注册表 |
| 世界与活动 | 农场地图、节日事件、市场、瀚海、旅行商人、钱包条目 | 建立地图、活动和交易内容注册表 |
| 纯规则或常量 | 时间推进、伤害、掉落倍率、结算顺序 | 保留在 engine/domain，不伪装成内容数据 |

盘点结果应形成机器可读清单，记录当前文件、导出符号、目标注册表、持久化 ID、引用方、迁移阶段和完成状态。只有清单中所有内容均已迁移或被明确判定为框架规则，才能宣告“官方内容包完全接管”。

### 6.9 Schema 单一事实来源

公开内容契约采用 TypeBox Schema 作为唯一事实来源。项目使用 `Static<typeof Schema>` 推导内部 TypeScript 类型，使用 Ajv 编译同一份 Schema 并校验外部 JSON，再从同一份 TypeBox 定义生成供模组作者和编辑器使用的公开 JSON Schema。建议依赖为 `@sinclair/typebox`、`ajv`，需要日期、URI 等标准格式时再引入 `ajv-formats`。

```text
TypeBox Schema
├─ Static<> 推导 TypeScript 类型
├─ Ajv 执行运行时结构校验
└─ 生成公开 JSON Schema
```

示例：

```ts
import { Static, Type } from '@sinclair/typebox'

export const LocalizedTextRefSchema = Type.Object({
  key: Type.String({ minLength: 1 }),
  fallback: Type.String({ minLength: 1 })
}, { additionalProperties: false })

export const ItemDefSchema = Type.Object({
  id: Type.String({ pattern: '^[a-z0-9_.-]+:[a-z0-9_./-]+$' }),
  name: LocalizedTextRefSchema,
  sellPrice: Type.Integer({ minimum: 0 })
}, { additionalProperties: false })

export type ItemDef = Static<typeof ItemDefSchema>
```

必须遵守以下约束：

- 不得为公开内容契约另行手写一套 TypeScript `interface` 或 `type`；类型必须从对应 TypeBox Schema 推导。
- 不得手工维护与 TypeBox 重复的公开 JSON Schema；导出的 `.schema.json` 文件只能由生成命令更新。
- 不得对外部数据使用 `parsedJson as ItemDef` 等断言绕过校验。外部值在 Ajv 校验成功前始终按 `unknown` 处理。
- TypeBox/Ajv 只负责字段、联合类型、格式和局部数值边界等结构校验；跨注册表引用、ID 所有权、依赖关系、标签传播和业务不变量由独立语义校验阶段负责。

完整挂载校验顺序为：

```text
签名与文件哈希校验
        → manifest Schema 校验
        → 旧 Schema 版本识别
        → 严格无损迁移
        → 当前版本 Ajv 结构校验
        → 包级暂存注册表
        → 跨注册表语义校验
        → 冻结 RegistrySnapshot
```

迁移函数的输入和输出都必须经过相应版本的 Schema 校验。结构校验失败不得进入语义校验；语义校验失败不得污染全局注册表。未来插件自定义注册表时，也必须提交可由挂载器编译的运行时 Schema，但是否允许定义公开注册表仍由后续机制插件的 `registry.define` 能力控制。

### 6.10 依赖与版本解析

拓扑排序只决定已经选定的包按什么顺序加载，版本解析必须先确定每个 `packageId` 唯一生效的版本。版本字段和依赖约束采用 SemVer 2.0；实现可使用成熟的 `node-semver`，公开契约首版支持确定版本、比较器范围、`^` 和 `~`，不支持 `latest`、Git URL、文件路径或脚本表达式。预发布版本只有在约束显式包含预发布标识时才参与匹配。

每个包文件的 `version` 必须是一个确定版本，一个包文件不得代表多个版本。同一个程序目录的单次启动环境中，同一 `packageId` 只允许一个生效版本；虽然依赖允许声明 SemVer 范围，但推荐模组作者依赖一个已经实际验证的确定版本。`mods/` 中存在同一包的多个版本且 lockfile 未明确选择时，挂载器必须停止解析并要求玩家处理，不得自动选择最新版。

所有必需依赖的版本约束必须同时满足。例如 A 要求 `library ^1.0.0`、B 要求 `library ^2.0.0` 时，不存在合法候选环境，挂载器必须在注册内容前阻止启动并展示完整依赖链。不得按目录扫描顺序、文件修改时间、最高版本或模糊优先级擅自消解冲突。

首版只把 `dependencies` 作为正式依赖类型：目标必须存在、版本必须满足，且目标包先加载。后续增加其他依赖类型时遵守以下语义：

- `optionalDependencies`：目标缺失时允许继续；目标已启用但版本不满足时仍视为冲突；存在且兼容时建立加载先后边。
- `loadBefore`、`loadAfter`：只提供确定性顺序提示，不代表可以引用目标内容，不参与版本选择，目标缺失时忽略，且不得覆盖必需依赖顺序。
- `peerDependencies`：数据插件阶段不提供。后续事件插件或机制插件确有共享运行库需求时再定义，不复用 npm 语义制造第二套嵌套依赖树。

游戏发行版本、引擎 API 版本和内容 Schema 版本是三个独立维度。较高版本游戏发现 `gameVersion` 较低的模组时，可以进入兼容尝试，但这不构成兼容保证，也不得绕过签名、权限、Schema、API 和语义校验：

```text
识别较低 gameVersion 的模组
        → 校验签名与文件完整性
        → 识别旧 contentSchemaVersion 并执行严格无损迁移
        → Ajv 结构校验
        → 包级暂存注册
        → API 与跨注册表语义校验
        → 成功后纳入候选 RegistrySnapshot
```

兼容尝试遇到任一错误时，应立即停止处理该模组，丢弃其全部暂存条目，并撤销依赖它的直接和传递依赖包。整个候选快照随之作废，`GameApp` 不得启动；挂载器继续运行并展示被放弃的模组、连带受影响包、失败阶段和错误日志。只有玩家明确禁用、移除或修复相应模组并重新解析后，才能产生新的候选环境。不得静默跳过失败模组后直接打开可能依赖其内容的存档。较低版本游戏不得尝试加载面向更高 `gameVersion`、更高引擎 API 或更高内容 Schema 的模组。

解析结果必须与文件扫描顺序无关。挂载器先扫描全部候选包，再按 `packageId` 分组并根据全局配置和 lockfile 选择唯一版本，检查版本约束后构建依赖图。稳定拓扑排序存在多个合法节点时，以规范化 `packageId` 字典序作为决胜规则。

每次完整挂载成功后，挂载器把最终包版本、内容哈希、依赖解析结果和稳定加载顺序原子写入程序目录 `userdata/mod-lock.json`。失败时不得覆盖上一次成功的 lockfile。lockfile 用于重现当前安装环境；存档仍须独立记录保存时实际使用的解析结果，用于环境比较和存档保护，不能反向控制全局模组启停。

### 6.11 生态规范与一致性契约

正式公开数据包 API 前，项目必须把生态约定拆成独立、版本化的规范，而不是只散落在实现代码和示例包中。至少发布清单、配置、本地化、生命周期和兼容性规范；规范版本独立于游戏版本、引擎 API 和内容 Schema 版本。

规范文本采用 RFC 风格关键词：

| 关键词 | 含义 |
| --- | --- |
| 必须（MUST） | 不满足即不兼容，校验器必须拒绝或给出规定错误 |
| 禁止（MUST NOT） | 出现即违反规范 |
| 应该（SHOULD） | 默认必须遵守，例外必须有可说明的理由 |
| 可以（MAY） | 可选能力，不得成为隐式依赖 |

每条可执行的强制规则都应具有稳定规则编号，例如 `PKG-LIFECYCLE-014`，并对应 Schema 或语义校验、稳定错误码以及至少一个一致性测试包。规范和实现发生分歧时，以已发布的稳定规范和测试结果为兼容承诺；修正文档不得静默改变既有稳定行为。

#### 6.11.1 配置作用域

模组全局启用状态与模组设置作用域分开。首个公开契约定义两种设置：

```ts
type PackageSettingScope = 'installation' | 'save'

interface PackageSettingDefinition {
  id: NamespacedId
  scope: PackageSettingScope
  default: unknown
  schema: unknown
}
```

| 作用域 | 保存位置 | 允许影响 |
| --- | --- | --- |
| `installation` | Electron 的 `userdata/mod-config/` 或平台等价存储 | 内容编译、注册表值、依赖解析和安装实例全局规则 |
| `save` | 根存档通用 `packageSettings` 区 | `RegistrySnapshot` 冻结后的当前存档运行时行为 |

会改变注册表条目、持久化 ID、配方、依赖关系、加载顺序或内容编译结果的设置必须使用 `installation`。这些内容相关配置的规范化摘要写入 `mod-lock.json` 和存档内容环境，防止相同包版本在不同配置下被误判为相同环境。`save` 设置不得参与包发现、版本选择、注册表定义或冻结前编译。

配置定义使用公开 Schema，挂载器根据布尔、枚举、数值、字符串等受支持类型生成标准控件；数据包不能借配置功能注入任意 Vue、HTML 或脚本。升级时保留玩家显式值，只为新增且缺失的字段补 Schema 默认值；配置迁移失败时保留旧值并拒绝候选提交，不得清空或静默重置。首阶段数据插件只实现 `installation` 配置，`save` 配置在事件或机制插件阶段启用，但根存档通用容器应提前定型。

主程序的纯显示偏好继续属于玩家设置，不伪装成模组配置。配置值不得包含密钥或依赖本方案提供秘密存储；未来确需敏感凭据时应另设平台安全存储能力。

#### 6.11.2 本地化

所有玩家可见的内容名称和描述使用稳定翻译引用，不把单一语言字符串作为唯一数据：

```ts
interface LocalizedTextRef {
  key: string
  fallback: string
}
```

语言标识采用 BCP 47，例如 `zh-CN`、`zh-TW`、`en-US`。翻译正文存放在清单声明的独立语言包中，翻译键必须位于数据包自己的命名空间。固定回退顺序为：

```text
当前地区语言
        → 同语言的默认地区
        → 数据包 defaultLocale
        → LocalizedTextRef.fallback
        → 显示翻译键并记录缺失诊断
```

默认语言和 `fallback` 必须存在。动态文本采用公开的 ICU MessageFormat 受限子集，并校验参数名称和复数分支；文本默认按纯文本渲染，禁止 HTML、脚本和事件属性。重复翻译键、非法语言标签、缺失默认语言和占位参数不一致必须进入文档或数据校验。官方内容开始迁移时即使用该结构，避免全量剥离后再次改写所有名称和描述。

#### 6.11.3 作者、许可与资源署名

公开清单支持 `authors`、`contributors`、`license`、`homepage`、`source`、`issues` 和 `attributions`。作者条目至少包含名称，可选角色和 HTTPS 地址；许可证优先使用 SPDX 标识。公共分发包必须声明许可证，本地开发包缺少许可证时可以加载但必须显示明确警告。

第三方图片、音乐、视频、字体和其他资源通过 `attributions.json` 逐项记录包内路径、作者、来源、许可证和是否修改。签名文件清单必须覆盖署名文件，构建和发布校验应检查资源路径存在且没有未署名的受管资源。挂载器只把外部链接交给系统浏览器，不在应用内直接执行或嵌入未知网页内容。

#### 6.11.4 原子生命周期

安装、升级、禁用和卸载使用统一事务状态机：

```text
discovered
        → staged
        → verified
        → resolved
        → mounted
        → committed
```

只有候选包、依赖图、配置迁移、资源和 `RegistrySnapshot` 全部通过后，才可以原子提交包文件、全局启用状态和 lockfile。Electron 在 `userdata/mod-transactions/` 保存崩溃恢复日志，Web 和 Android 使用平台等价事务存储；启动时必须先恢复或回滚未完成事务，再扫描模组。

- 安装失败不得在正式 `mods/` 中留下半个包。
- 升级先保留旧版本并暂存新版本，失败时继续使用旧版本和旧 lockfile。
- 禁用先验证依赖它的包和候选环境，成功后才更新全局配置。
- 卸载必须先完成禁用事务，默认只移除包文件，保留配置、`pluginData`、存档未知内容和可恢复备份。
- “同时清除插件数据”是独立危险操作，必须明确确认并先备份受影响存档。
- 安装和升级不得批量改写现有存档；存档迁移只在玩家打开副本时发生。
- 已运行的 `GameApp` 不热切换模组，生命周期变更在重新挂载后生效。

文件系统原子替换、IndexedDB 事务和 Android 存储操作由平台适配器实现，但共享同一状态机、错误码和回滚保证。

#### 6.11.5 弃用与兼容测试包

公开契约按 `experimental → stable → deprecated → removed` 管理。`experimental` 能力不作长期兼容保证；`stable` 能力只能按已发布版本策略演进。普通弃用至少保留两个稳定发布周期，并提供 `deprecatedSince`、替代接口、迁移示例、计划移除版本、挂载器警告和稳定错误码。移除稳定 API 原则上只能发生在新的 `engineApiVersion` 主版本；安全漏洞可以紧急禁用，但必须发布安全公告和明确诊断。

弃用清单同时以人类可读变更日志和机器可读索引发布。兼容承诺只覆盖正式公开的 Schema、注册表、事件和能力 API，不覆盖项目内部 TypeScript 模块。

项目维护公开的一致性测试包集合：

```text
compatibility-packs/
  valid/       # 最小合法包、全部注册表、配置、本地化和资源署名
  invalid/     # 缺失依赖、版本冲突、重复 ID、非法配置和越权资源
  legacy/      # 旧 Schema、弃用 API、旧本地化和迁移链
```

每个测试包包含预期结果，至少校验成功或失败、错误码、修复报告、最终加载顺序、lockfile、注册表快照哈希、配置摘要、缺失插件数据保留和生命周期失败后的目录状态。项目发布 CI 和模组作者的独立校验命令必须复用同一组夹具；只有通过对应一致性测试的能力才能标记为 `stable`。
