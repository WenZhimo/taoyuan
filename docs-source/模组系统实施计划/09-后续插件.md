# 后续事件插件与机制插件

> [返回统筹说明](./README.md)

## 16. 后续：事件插件

数据插件稳定后，可以在不暴露 Store 的前提下增加类型化事件系统。首批候选事件：

- `onMonsterDefeated`
- `onItemUse`
- `onDayEnd`
- `onCropHarvest`
- `onDamageCalculate`

事件系统应区分通知事件、可取消事件和结果变换管线。插件接收只读上下文，返回经过引擎校验的命令或结果补丁；不得直接修改 Pinia Store。还必须定义优先级、稳定执行顺序、异常隔离、随机源、执行时间限制和插件专属存档空间。

首版事件插件优先采用声明式条件与动作，例如“击败某怪物后按概率发放某物品”。任意 JavaScript 执行不应与数据包一起默认开放。

事件插件如需保存只供自身处理器读取的可变运行时状态，应使用按 `PackageId` 隔离的 `pluginData`；如需保存启动时注册、可从插件包重建的结构化定义，才申请插件私有注册表。私有注册表仍必须提供 Schema、版本和存档缺失策略，但默认不允许其他插件引用，也不会自动获得 UI、路由或游戏系统消费者。将私有注册表升级为跨插件公开契约时，应按机制插件要求重新审核。

## 17. 后续：机制插件

事件系统稳定后，可以逐步提供更高级的机制扩展点：

- 新副本：复用通用地图、遭遇、掉落和结算框架。
- 新设施：通过设施定义、建造条件、输入输出槽和周期处理器扩展。
- 新 UI 页面：优先使用宿主提供的声明式页面和标准组件。
- 新系统规则：通过明确的规则服务、策略接口和能力授权扩展。

复杂机制不应直接获得整个游戏对象。推荐按能力开放，例如“读取背包快照”“请求发放物品”“注册导航入口”，并由引擎校验调用。

机制插件可以申请 `registry.define` 能力，在注册表类型目录冻结前定义自有公开注册表。申请方必须同时提供数据消费者、Schema、版本策略、跨插件引用规则、存档兼容和禁用恢复行为；仅声明一个无人读取的数据集合不构成开放公开注册表的理由。

机制插件的静态定义进入自定义注册表，存档中的可变实例状态进入自身 `pluginData`。机制插件不得要求游戏本体为其新增根存档字段，也不得直接访问其他插件的私有存档区。

如未来确实需要受信任代码插件，应将其限定为桌面端高级功能，使用独立进程或专门沙箱、权限清单、超时和崩溃隔离。不得通过开启 Electron `nodeIntegration` 或直接执行未知脚本来实现。

## 18. 建议的首个实施切片

首个开发切片建议只完成以下闭环：

1. 建立 `TagRegistry`、`ItemRegistry`、`RecipeRegistry` 和内容包清单。
2. 抽取少量官方物品、标签和“包子”配方作为试点，其余内容继续经适配器注册。
3. 实现标签材料的匹配、分配、选择、品质计算和原子扣料闭环。
4. 实现组成标签传播、普通标签合堆取并集，以及 `taoyuan:human_meat` 等特殊标签强制分堆。
5. 提供一个外部示例包，新增一种带官方标签的普通物品、一条标签配方和一个现有商店商品。
6. 验证背包显示、购买、标签选料、批量烹饪、堆叠/拆分、保存、读取和缺失数据包占位。
7. 证明流程成立后，再批量迁移完整物品和配方数据。

这个切片足以暴露注册表、引用、平台加载、UI 和存档问题，同时不会一开始就触碰矿洞战斗和隔夜结算等高风险流程。
