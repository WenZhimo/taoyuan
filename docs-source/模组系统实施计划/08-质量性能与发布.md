## 11. 测试与质量门禁

测试策略遵守 [测试与质量门禁指南](../testing-guide.md)。每阶段至少包括：

| 测试类型 | 重点 |
| --- | --- |
| Registry 单元测试 | 注册表定义权限、组合键重复、双阶段冻结、查询、来源追踪、稳定顺序 |
| Schema 测试 | 全部 TypeBox Schema 可由 Ajv 编译；缺失字段、未知字段、非法数值和版本不兼容会被拒绝 |
| Schema 生成测试 | 官方内容通过导出的公开 Schema；文档 JSON 示例可校验；生成结果与已提交文件一致，CI 在生成后出现差异时失败 |
| 规范一致性测试 | 每条稳定 MUST/MUST NOT 规则具有规则编号、对应错误码、机器校验和至少一个合法或非法夹具 |
| 语义校验测试 | 跨注册表引用、ID 所有权、依赖、标签传播和业务不变量；确认结构校验与语义校验相互独立 |
| 自动修复测试 | 白名单规则幂等、修复报告、禁止有损修复、修复后完整复验、签名原文件不改写 |
| 依赖解析测试 | SemVer 范围、确定版本优先建议、单包单版本、重复版本候选、无解版本冲突、缺失依赖、循环依赖、扫描顺序无关和稳定拓扑排序 |
| 旧模组兼容测试 | 高版本游戏尝试低 `gameVersion` 模组；成功时完整挂载，失败时撤销该模组及传递依赖、保留日志且不创建 `GameApp` |
| Lockfile 测试 | 成功挂载后原子更新版本、哈希和顺序；失败不覆盖上次结果；存档环境与 lockfile 职责分离 |
| 注册表缓存测试 | 相同环境命中、完整挂载等价、环境变化失效、损坏回退、原子写入、索引重建、清理不影响存档和官方预编译三端一致 |
| 配置测试 | 安装级与存档级作用域隔离、默认值、显式值保留、配置迁移、内容配置哈希、缺失包保留和禁止存档配置改变注册表 |
| 本地化测试 | BCP 47、默认语言、固定回退链、ICU 参数一致性、缺失键诊断、纯文本渲染和官方内容翻译引用覆盖 |
| 发布元数据测试 | 作者字段、SPDX 许可证、HTTPS 链接、资源路径与逐项署名完整性 |
| 生命周期事务测试 | 安装、升级、禁用和卸载各阶段故障注入、崩溃恢复、旧版本回退、配置与 lockfile 一致性以及默认保留私有数据 |
| 弃用与兼容包测试 | `experimental/stable/deprecated/removed` 状态、弃用警告、迁移入口，以及 `valid/invalid/legacy` 夹具的固定结果 |
| 等价性快照 | 迁移前后官方内容 ID、数值、引用和排序一致 |
| Store 回归测试 | 背包、配方、商店、战斗和存档读取 |
| 材料分配测试 | 重叠标签、择一标签、品质、批量制作、原子扣料和稳定选料顺序 |
| 制品堆叠测试 | 普通标签并集、来源丢失、特殊标签分堆、拆分、旧存档迁移和行为敏感标签校验 |
| 缺失内容测试 | 禁用数据包后保留未知状态，重新启用后恢复 |
| 插件私有存档测试 | 缺失、禁用和重新启用插件时信封字段、`payloadJson` UTF-8 内容及哈希保持不变；所有权隔离、Schema 迁移、单包与总量配额、超限拒绝和原子回滚 |
| 版本兼容测试 | 原版逐版本升级、降级拒绝、模组环境不匹配、内容哈希变化、跨目录副本和失败回滚 |
| 性能测试 | 10 万定义注册、索引、引用扫描、冷挂载与缓存恢复耗时、缓存大小和列表分页 |
| 挂载器测试 | 全局配置、安全模式、进度、失败不启动游戏本体、快照冻结 |
| 安全测试 | 路径穿越、重解析点、ZIP 炸弹、超大 JSON、MIME 欺骗、签名篡改和未受信任可执行文件 |
| 桌面烟雾测试 | 实际启动 `pkg/win-unpacked/taoyuan.exe`，验证挂载流程、数据目录及重新构建后 `userdata/`、`mods/` 保留 |

最终门禁：

```bash
pnpm run type-check
pnpm run lint
pnpm run generate:schemas
pnpm run validate:schemas
pnpm run validate:data
pnpm run validate:docs
pnpm test
pnpm run build
git diff --check
```

涉及 Electron 加载器或目录行为时额外执行：

```bash
pnpm run build:electron
```

并实际启动未压缩版验证数据包发现、错误提示、存档写入位置和日志。

## 12. 性能和资源限制

首版应设置可解释、可配置的防护限制，不用静默截断掩盖问题：

- 限制单个压缩包和解压后总大小，并在拒绝时说明原因。
- 限制单个 JSON 文件的深度和条目数，避免解析阻塞和内存耗尽。
- 限制语言包、配置 Schema、资源署名清单的大小和条目数，并在挂载器中报告对应包和文件。
- 限制单个插件私有存档和全部插件私有存档的未压缩 UTF-8 字节数、深度、节点数和单次增长量；超限时拒绝候选写入并保留旧信封。
- 大数据包校验应在挂载器中分阶段报告进度；JSON 解析、Schema 校验和引用扫描优先放入 Worker/受控后台任务，不能仅依靠循环中插入 `setTimeout` 掩盖单次阻塞。
- 注册表查询使用 Map；分类、商店和区域池建立一次性索引。
- 资源文件按需加载，不在启动时解码全部图片和音频。
- 注册表缓存必须绑定完整 `environmentHash`，包括包内容、安装级配置、Schema、编译器、加载顺序和信任策略；命中时仍需验证缓存结构和快照哈希。
- 缓存恢复只重建运行时 Map、索引和只读包装，不重新执行完整外部 JSON 校验；任一恢复不变量失败自动退回完整挂载。

## 13. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 迁移改变既有数值或排序 | 玩家可见行为回归 | 规范化快照和固定随机测试 |
| 页面继续直接导入内置数组 | 第三方内容在部分界面不可见 | 全仓搜索、兼容入口统一访问注册表 |
| ID 命名空间破坏旧存档 | 内容丢失或无法读取 | 双读、别名映射、占位保留 |
| 新版存档被旧版写坏 | 不可逆数据损失 | 版本目录隔离、禁止降级、跨版本只迁移副本 |
| 玩家误以为模组存档保证升级 | 更新后无法读取 | 挂载器显示精确环境差异，发布文档明确第三方迁移不受官方保证 |
| 数据包依赖形成循环 | 无法确定加载顺序 | 显式依赖图和循环诊断 |
| 同一依赖的版本约束无交集 | 无法形成合法候选环境 | 单包单版本、SemVer 约束求交并在注册前阻止启动 |
| 文件扫描顺序改变模组结果 | 同一目录出现不可重现行为 | lockfile 固定选择，稳定拓扑排序以规范化包 ID 决胜 |
| 旧模组在新游戏中部分加载 | 注册表污染或存档缺失内容 | 仅在暂存区兼容尝试，失败撤销传递依赖并停止游戏本体启动 |
| 过期或篡改缓存被当作有效注册表 | 加载旧内容、绕过校验或产生错误引用 | 完整环境键、版本化 Schema、快照哈希、恢复不变量和自动完整挂载回退 |
| 缓存格式绑定 Node/Electron 实现 | 升级运行时后无法恢复或跨平台结果不同 | 使用显式版本化 JSON/CBOR codec，禁止长期依赖 `v8.serialize()` |
| 安装级与存档级配置混用 | 不同存档改变冻结注册表或环境无法重现 | 设置 Schema 声明作用域，内容相关配置只允许安装级并写入环境哈希 |
| 本地化在内容迁移后补做 | 官方内容需要再次全量改写且第三方字段不一致 | 阶段 0 冻结翻译引用和回退规则，首个试点即使用语言包 |
| 模组生命周期操作非原子 | 半安装、升级失败或 lockfile 与文件不一致 | 统一事务状态机、候选快照、崩溃日志和上一版本回退 |
| 稳定 API 无预告移除 | 大量模组在游戏更新后同时失效 | 生命周期状态、两个稳定周期弃用窗口、机器警告和兼容夹具 |
| 第三方资源缺少许可或署名 | 发布争议和资源无法追溯 | SPDX 元数据、逐资源署名清单和发布校验 |
| 自定义机制直接扩展根存档 | 核心存档随插件类型持续膨胀且缺失插件时无法解释 | 统一 `pluginData` 不透明信封，插件自行提供 Schema 和迁移 |
| 插件私有数据失控增长 | 存档膨胀、解析阻塞或内存耗尽 | 单包与总量配额、结构限制、候选写入校验且禁止静默截断 |
| 外部压缩包包含恶意路径或超大文件 | 文件覆盖、内存或磁盘耗尽 | 路径规范化、大小和数量限制 |
| 重新打包清除第三方模组 | 玩家环境丢失 | 构建前后同步备份、恢复并校验 `mods/` |
| 签名被误认为安全审计 | 受信任发布者代码仍可能越权 | 签名只证明身份和完整性，执行仍受沙箱与能力系统限制 |
| 附魔或条件表达能力过弱 | 数据作者转向硬编码请求 | 先提供标准效果和条件集合，再按真实需求扩展 |
| 数据 DSL 过度复杂 | 实际变成难维护的编程语言 | 保持数据插件声明式，行为交给后续事件 API |
| 官方内容与引擎版本耦合 | 更新后旧数据包失效 | 独立 API/Schema 版本和弃用周期 |

## 14. 交付物清单

1. 内容 API、注册表架构和版本化生态规范，包括配置、本地化、发布元数据、生命周期、弃用和错误码契约。
2. 全部官方静态内容的注册表迁移清单，以及作为单一事实来源的 TypeBox Schema、由 `Static<>` 推导的 TypeScript 类型和生成的只读公开 JSON Schema。
3. `taoyuan-core` 官方内容包。
4. 内容编译器、确定性依赖解析器、`mod-lock.json` Schema、持久化注册表快照 codec/cache 和统一校验器。
5. Electron、Web、Android 三种内容来源适配器。
6. 独立挂载器、原子生命周期事务、全局模组管理、发布者信任库和错误诊断界面。
7. 存档内容环境、通用 `packageSettings`、未知内容保留、插件私有存档区和安全模式。
8. 最小示例包、综合示例包、`valid/invalid/legacy` 一致性测试包、Schema 生成命令和独立校验命令。
9. 等价性、规范一致性、兼容性、缓存恢复、生命周期、性能和安全回归测试。

## 15. 提交与发布策略

- 每个阶段拆成单一职责提交，不把内容迁移与玩法调整混在一起。
- 注册表基础、每个内容域迁移、平台加载器、存档迁移和 UI 管理分别提交。
- 在官方内容完全切换前保留兼容适配层；确认一个发布周期稳定后再删除旧读取路径。
- 外载数据包能力先作为实验功能发布，仅承诺当前 `engineApiVersion`。
- 官方只保证原版旧存档向原版新版本升级；模组存档跨游戏版本或模组版本不作兼容保证。
- Electron 发布说明应建议不同游戏版本放在不同目录，分别维护 `mods/` 和 `userdata/`；跨版本存档只能复制导入。
- 不提供新版存档向旧版游戏的降级写入路径。
- 数据包加载失败不得阻止安全模式启动，也不得自动覆盖原存档。
- 首次正式开放前至少使用一个非官方示例包走完安装、创建存档、升级数据包、缺失数据包和恢复流程。
