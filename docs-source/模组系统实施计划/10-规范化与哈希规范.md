# 规范化与哈希规范

> [返回统筹说明](./README.md)

## 目的

内容环境、配置、lockfile、签名和插件私有数据都依赖哈希。所有平台必须对相同逻辑输入产生相同结果，不能依赖对象键顺序、文件扫描顺序或平台路径格式。

## 通用格式

- 哈希算法统一使用 SHA-256。
- 文本表示统一为 `sha256:` 加 64 位小写十六进制，例如 `sha256:0123...abcd`。
- JSON 规范化采用 RFC 8785 JSON Canonicalization Scheme（JCS），编码为无 BOM 的 UTF-8。
- JCS 不替业务数据执行 Unicode 归一化；持久化 ID 本身限制为 ASCII，玩家可见文本按原始 Unicode 内容参与哈希。
- 包内路径先把 `\` 转为 `/`，拒绝绝对路径、`.`、`..`、空段、NUL 和重解析点，再按 Unicode NFC 规范化。
- 所有排序使用 Unicode 码点升序；不得使用当前系统区域设置或文件系统返回顺序。

## 文件完整性

单文件哈希针对原始文件字节，不转换换行符、不移除 BOM、不重编码媒体。规范化只作用于文件清单中的相对路径表示，不改变被签名文件。

规范化文件清单条目为：

```ts
interface CanonicalFileEntry {
  path: string
  size: number
  sha256: string
}
```

条目按 `path` 排序后以 JCS 编码。签名文件本身不得递归进入被签名清单；具体排除项必须由清单 Schema 固定，不允许包作者自定义。目录包和 ZIP 包解出相同文件时必须产生相同清单哈希。

## 内容包哈希

`contentHash` 是规范化文件清单的 SHA-256，覆盖清单、数据入口、语言包、资源和署名文件。它表示发布包的精确内容，不等同于版本号。

```text
原始文件字节 → 每文件 SHA-256
        → 规范化相对路径和大小
        → 按路径稳定排序
        → JCS 编码文件清单
        → SHA-256 = contentHash
```

签名验证先于内容解析。任何文件缺失、额外文件、大小变化或哈希变化都使签名和 `contentHash` 失效。

## 配置哈希

`configurationHash` 只覆盖会影响内容编译、依赖或注册表结果的安装级配置。输入对象包含配置 Schema 版本和规范化值：

```json
{
  "schemaVersion": "1",
  "values": {}
}
```

缺少内容相关配置时仍计算上述空对象的固定哈希，不能使用空字符串或省略字段。存档级配置、纯 UI 偏好和敏感凭据不得进入 `configurationHash`。

## 注册表快照哈希

`RegistrySnapshot` 哈希用于测试、诊断和未来缓存完整性校验，不替代包签名。当前 v2 哈希正文由 `formatVersion` 和 `registries` 组成，不包含 `snapshotHash` 自身：

- `registries` 使用 `RegistrySet.registryIds()` 的稳定 ID 顺序。
- 每个注册表的 `entries` 数组保留真实注册顺序，不按 `contentId` 排序；该顺序同时决定恢复后的 `entries()` 和 `values()`。
- JCS 继续只按码点规范化对象键，不改变数组顺序。因此内容、owner/source 完全相同但注册顺序不同的两个快照必须产生不同哈希。
- 来源只允许包 ID、包内相对路径和本地 ID；绝对路径、加载耗时、内存地址、本地化后的当前显示文本和缓存命中字段不得进入正文。

v1 曾在生成快照时按 `contentId` 排序，无法表达完整构建的原始注册顺序。它是可重建的派生缓存格式，只能失效并重新生成，不能通过补写元数据伪装成 v2，也不涉及玩家存档迁移。当前官方 v2 快照含 54 个注册表、4242 条目，哈希为 `sha256:4e87e4bc1d6310d4467335da77603006bf769fdb5c4da45ad927f7ed85a5c4b3`。

派生内容必须在固定输入下稳定生成。随机派生不得发生在内容编译阶段；确需随机的运行时系统必须使用显式种子和稳定算法版本。

## Lockfile 哈希

lockfile 的候选内容先按稳定加载顺序排列，每个包至少记录：

```text
PackageId
version
contentHash
configurationHash
loadIndex
resolvedDependencies
```

`resolvedDependencies` 按依赖包 ID 排序。lockfile 自身的摘要字段不参与自身哈希。只有整个候选环境成功后才原子替换正式 lockfile。

## 缓存环境哈希

注册表缓存的 `environmentHash` 使用 JCS 编码以下逻辑对象后计算 SHA-256：

```ts
interface CacheEnvironmentIdentity {
  gameVersion: string
  engineApiVersion: string
  contentSchemaVersion: string
  loaderVersion: string
  contentCompilerVersion: string
  schemaSetHash: string
  cacheFormatVersion: number
  trustPolicyVersion: string
  packages: Array<{
    id: PackageId
    version: string
    contentHash: string
    configurationHash: string
    loadIndex: number
    resolvedDependencies: PackageId[]
  }>
}
```

`packages` 按 `loadIndex` 排序，依赖 ID 按码点排序。时间戳、绝对路径、缓存目录、加载耗时和当前语言不得进入环境哈希。缓存正文另行计算 `snapshotHash`；只有环境哈希和快照哈希都匹配时才能恢复。详细流程见 [注册表快照缓存](./15-注册表快照缓存.md)。

## 插件私有负载

`pluginData.payloadHash` 直接针对 `payloadJson` 字符串的 UTF-8 字节，不执行 JCS。这样缺失插件时核心能够验证并原样搬运不透明负载。插件生成新负载时应使用 JCS，便于自身获得稳定结果，但核心不得在插件缺失时替它重新规范化。

## 一致性测试

至少提供以下黄金向量：

- 不同对象键顺序产生相同 JCS 哈希。
- Windows 和 POSIX 路径表示产生相同规范化清单。
- CRLF 和 LF 文件作为原始字节时产生不同文件哈希。
- 目录包和等价 ZIP 包产生相同 `contentHash`。
- 文件扫描顺序变化不得意外改变内容注册顺序；有意改变注册顺序必须改变快照哈希，配置、文件清单和 lockfile 仍按各自规范排序。
- 快照 `create -> restore -> create` 保持正文和哈希完全相同，仅调整任一注册表的条目顺序会改变哈希。
- 相同逻辑环境在三端产生相同 `environmentHash`，任一包、配置、Schema、编译器或信任策略变化都会改变它。
- `payloadJson` 任意字节变化都会使 `payloadHash` 失效。
- Node、Electron、浏览器和 Android WebView 对官方测试向量输出一致。
