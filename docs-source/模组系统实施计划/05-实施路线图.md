## 8. 分阶段实施计划

### 阶段 0：基线、决策记录与等价性夹具

#### 目标

在改变数据来源前固定当前行为，建立可以判断“迁移是否改变内容”的基线。

#### 任务

1. 记录内容 API、命名空间、冲突策略和平台加载边界。
2. 导出当前核心目录的规范化快照：ID、关键数值、引用关系和排序。
3. 补齐商店商品、怪物池、附魔权重和派生物品的校验测试。
4. 建立测试用旧存档，覆盖背包物品、已解锁配方、附魔武器和矿洞状态。
5. 明确现有裸 ID 到 `taoyuan:*` 的别名迁移策略。
6. 明确标签材料的匹配、手动选择、自动选择、批量分配、品质计算、组成标签传播、特殊标签分堆和原子扣料语义。
7. 逐文件盘点 `src/data/` 的全部导出，形成机器可读的注册表迁移清单，并区分内容定义、派生数据和框架算法。
8. 建立版本化模组规范目录、RFC 关键词、稳定规则编号和错误码登记方式，并在迁移官方内容前冻结 `LocalizedTextRef`、BCP 47 语言包和发布元数据的首版契约。

#### 验收标准

- 不改变任何运行时代码和玩法数值。
- 快照和旧存档夹具可在后续阶段重复使用。
- `type-check`、Lint、数据校验、文档校验、完整测试和 Web 构建通过。

#### 回滚点

本阶段只增加决策记录和测试，可独立提交和回滚。

### 阶段 1：注册表与内容契约

#### 目标

建立通用注册表、内容 ID、清单类型、校验错误模型和冻结生命周期，但暂不迁移内容来源。

#### 任务

1. 实现 `ContentId`、`PackageId` 和命名空间解析。
2. 实现可记录来源的通用 `Registry<T>`，以 `(registryId, contentId)` 作为条目完整身份。
3. 实现按 `RegistryTypeId` 查询的通用 `RegistrySet` 和启动阶段状态机，不把内置注册表写成固定属性。
4. 引入 TypeBox 与 Ajv，以 TypeBox Schema 作为公开内容契约的唯一事实来源；通过 `Static<>` 推导内部类型，并生成禁止手工编辑的公开 JSON Schema。
5. 为首批七类注册表定义 TypeBox Schema；配方材料契约同时支持固定物品、单标签和多标签择一。
6. 将 Ajv 结构校验与跨注册表语义校验拆为独立阶段，并实现重复 ID、未知引用、数值边界和依赖错误模型。
7. 为现有静态数组建立临时适配器，将其注册到新注册表。
8. 为标签定义增加传播和堆叠策略，为运行时物品栈增加可选 `compositionTags`，并定义旧存档缺失字段的默认迁移。
9. 为清单、本地化引用、配置定义、作者元数据和资源署名建立 TypeBox Schema、错误码和最小合法/非法夹具。
10. 将纯数据的持久化注册表快照契约与包含 Map、索引和只读包装的运行时 `RegistrySnapshot` 分离，定义版本化 codec 和快照哈希。

#### 验收标准

- 游戏仍使用现有内容，页面行为不变。
- 注册表可以拒绝重复 ID，并在冻结后拒绝注册。
- 不同注册表可以保存相同 `contentId`；同一注册表内的重复完整 ID 会被拒绝。
- 第一阶段的数据包不能定义新注册表，但底层目录无需修改核心结构即可在后续接纳授权注册表。
- 所有 TypeBox Schema 均可被 Ajv 编译，内部公开契约类型均由 `Static<>` 推导，生成的公开 JSON Schema 与源定义一致。
- 结构错误在进入暂存注册表前被拒绝，跨注册表和业务错误在独立语义校验阶段被拒绝。
- 错误包含数据包、文件、字段路径和内容 ID。
- 旧的 `itemId + quantity` 配方可以标准化为固定物品材料，标签和材料引用在注册表冻结前完成校验。
- 行为敏感标签不能使用 `stackPolicy: merge`；旧物品栈缺少 `compositionTags` 时稳定迁移为空数组。
- 官方试点内容使用翻译键和 `fallback`，清单元数据、语言包和资源署名可被统一校验。
- 持久化快照不包含函数、响应式对象、Store、绝对路径和资源句柄；编码再恢复后查询结果和 `snapshotHash` 不变。
- 对 10 万条测试定义完成注册和查找时，界面启动流程不被长时间阻塞；热路径查找使用 Map，不再使用重复线性扫描。

#### 回滚点

保留原查找函数签名，适配器失败时可以恢复原静态实现。

### 阶段 2：物品、作物和配方试点

#### 目标

完成第一条端到端官方内容包路径，验证注册、派生、引用、UI 和存档。

#### 任务

1. 抽取物品、作物和烹饪配方到 `taoyuan-core`。
2. 建立内容编译器，生成种子、收获物和烹饪成品。
3. 将 `getItemById()`、`getRecipeById()` 等入口改为访问注册表。
4. 保持现有裸 ID 兼容，并在内部规范化为官方命名空间。
5. 实现统一材料分配器，并让可制作判断、最大数量、品质预览和实际扣料复用同一规划结果。
6. 在烹饪弹窗中展示标签槽的自动选择结果，并允许玩家更换具体材料。
7. 以“面粉 + 任意素食或肉类”的包子配方作为标签材料试点。
8. 实现制品组成标签传播和新堆叠签名：普通标签合并取并集并丢弃来源，特殊标签不同则禁止合堆。
9. 覆盖背包、仓库、送礼、烹饪、商店出售和存档导入流程。

#### 验收标准

- 迁移前后物品和配方规范化快照一致。
- 所有现有物品名称、售价、分类、恢复值和配方材料不变。
- 包子可以使用任一符合标签的材料；同一库存项不会在重叠标签槽之间重复计数或重复扣除。
- 单份和批量制作的可制作判断、最大数量、选料、成品品质和实际扣料保持一致，失败时库存不发生部分变化。
- 普通馅料包子可以合堆且合并标签后不保存逐份来源；含 `taoyuan:human_meat` 的包子不能与不含该标签的包子合堆。
- 合堆、拆分、保存、读取和旧存档迁移都保持相同的特殊标签隔离结果。
- 旧存档可以加载、保存并再次加载。
- 未知物品使用占位定义保留原 ID、数量、品质和组成标签，不被过滤或清零。

#### 回滚点

按内容域保留旧数据导出一个发布周期；注册表入口可以切回静态适配器。

### 阶段 3：商店商品迁移

#### 目标

消除页面内按商店硬编码商品列表的情况，使第三方商品可以自然出现在现有商店 UI。

#### 任务

1. 定义 `ShopDef`、`ShopOfferDef` 和显示分组。
2. 抽取种子、武器、帽子、鞋、材料和特殊商品为统一商品定义。
3. 将季节、天气、进度、库存等条件表达为受支持的声明式条件。
4. 让 `ShopView.vue` 只渲染注册表返回的分组和商品。
5. 保留现有购买动作、数量上限、音效和日志流程。

#### 验收标准

- 六个现有商店的营业状态、商品、价格和解锁条件不变。
- 新增商品不需要修改 `ShopView.vue` 分支。
- 失效商品引用在启动阶段报错，不在玩家点击购买时才报错。

### 阶段 4：附魔和掉落表迁移

#### 目标

把可配置的标准效果与游戏算法分开，并建立统一命名掉落表。

#### 任务

1. 抽取附魔定义、稀有度、权重和标准效果参数。
2. 将附魔效果限制为引擎已支持的效果类型。
3. 把怪物、宝箱、采集和其他可统一的掉落条目迁移到 `DropTableRegistry`。
4. 保留现有超过 100% 的掉落倍率算法和可注入随机源。
5. 建立掉落表嵌套、条件、权重和数量范围的校验规则。

#### 验收标准

- 现有附魔属性、显示、价格、随机权重和战斗效果不变。
- 固定随机输入下，迁移前后掉落结果一致。
- 数据插件可以定义新的标准数值附魔；新的未知机制必须被拒绝，而不是静默忽略。

### 阶段 5：怪物与矿洞内容迁移

> 状态：已完成当前范围总验收。怪物定义和 15 个官方有序怪物池已迁移，矿洞生成、缩放、战斗与随机算法继续留在框架。

#### 目标

将怪物和怪物池配置抽离，同时保持矿洞生成和战斗算法属于框架。

#### 任务

1. 抽取普通怪物、BOSS、区域怪物池和骷髅矿穴怪物池，分别注册到 `taoyuan:monster` 和 `taoyuan:monster_pool`。
2. 怪物通过掉落表 ID 引用掉落，不再内嵌重复掉落结构。
3. 矿洞生成器从注册表解析区域池，不直接导入静态对象。
4. 对缺失怪物池、空池和无效 BOSS 配置提供启动诊断。
5. 保持怪物击败、成就、行会记录、经验和战利品结算顺序不变。

#### 验收标准

- 固定随机输入下，楼层怪物分布和战斗结算与迁移前一致。
- 怪物池解析不消耗随机数，原随机调用次数和调用位置保持不变。
- 当前矿洞、骷髅矿穴和 BOSS 流程通过回归测试。
- 单纯新增符合现有战斗模型的怪物不需要修改 Store 或 Vue 页面。

### 阶段 7：外载数据包加载器

#### 目标

允许游戏发现、导入、校验和加载第三方数据包。

#### 任务

1. 定义目录包和 ZIP 包的安全读取规则。
2. 实现包级暂存注册表和候选快照，确保校验失败时可以丢弃整个包且全局注册表从未进入部分提交状态。
3. 实现严格白名单自动修复器和结构化 `RepairReport`；修复只修改暂存区规范化结果，不改写原始包文件。
4. 实现单包单版本选择、SemVer 依赖约束求交、版本冲突诊断、循环依赖检查和稳定拓扑排序，确保文件扫描顺序不影响结果。
5. 实现高版本游戏对低 `gameVersion` 模组的暂存兼容尝试；任一错误立即撤销该模组及其直接和传递依赖，并终止本次 `GameApp` 启动。
6. 实现资源路径解析，禁止绝对路径和越过数据包根目录的相对路径。
7. Electron 主进程只负责文件发现和读取，通过受限 IPC 返回数据；不得打开 `nodeIntegration`。
8. Web 使用文件选择器导入并保存到 IndexedDB。
9. Android 使用系统文件选择器导入到应用数据目录。
10. 三个平台都调用同一个解析、修复、校验和注册核心。
11. 将官方 `content-packs/taoyuan-core` 明确加入 Vite、Electron Builder 和 Android 构建输入，校验三端发布产物中的版本与哈希一致。
12. Electron 构建脚本在清理 `pkg/` 前同时备份 `win-unpacked/userdata/` 和 `win-unpacked/mods/`，打包后恢复并核对文件数、总字节数和内容哈希；失败时保留安全备份并阻止脚本报告成功。
13. 实现跨平台统一的安装、升级、禁用和卸载事务状态机；Electron 写入崩溃恢复日志，其他平台使用等价事务机制，任何阶段失败都恢复包文件、全局配置和 lockfile 的上一致状态。
14. 实现以 `environmentHash` 为键的持久化注册表快照缓存；命中时恢复纯数据条目并重建运行时索引，未命中、损坏或超限时自动完整挂载并原子更新缓存。

#### 验收标准

- 可以安装一个仅含新物品、配方和商店商品的示例数据包。
- 缺失依赖、版本不兼容、重复 ID、非法路径和错误引用均有可理解提示。
- 同一包的多个版本不会同时生效；互不兼容的 SemVer 约束会在注册前阻止启动，并且重复扫描同一组文件始终得到相同解析结果。
- 较高版本游戏可以尝试暂存加载低版本模组；兼容失败时该模组及依赖它的包全部撤销，挂载器保留诊断但游戏本体不启动。
- 安装、升级、禁用和卸载在注入失败或进程中断后不会留下半安装包、半更新配置或与包文件不一致的 lockfile。
- 相同环境第二次启动命中快照缓存，恢复结果与完整挂载完全一致；修改任一包、配置、Schema、编译器或信任策略都会失效。
- 第三方数据包不能读取任意本地文件，也不能获得 Electron 或浏览器执行权限。
- 损坏数据包被完整撤销，不会污染官方内容包或其他包的暂存结果；当前启动流程仍按下一条要求终止。
- 任一启用包无法修复或校验失败时，`GameApp` 不会创建，错误日志包含修复尝试和回滚结果，全局注册表不存在残留条目。
- 重新执行 `build-exe.bat` 后，玩家现有 `userdata/` 和 `mods/` 均完整保留，官方内容包随新版本正确更新。

### 阶段 8：模组管理与存档保护

#### 目标

通过独立挂载器让玩家理解全局启用了什么，在进入游戏本体前完成挂载，并在内容缺失时保护存档。

#### 任务

1. 增加 `LauncherApp`，展示数据包列表、全局启用状态、版本、依赖、信任状态、加载进度和错误详情。
2. 挂载器读取并保存全局模组配置；数据包启停后重新构建注册表快照，不直接修改任何存档；成功后原子更新 `userdata/mod-lock.json`，失败时保留上一次成功结果。
3. 在存档根对象增加统一 `saveFormatVersion` 和内容环境元数据，包括游戏版本、引擎 API、内容 Schema、数据包 ID、版本、内容哈希和稳定加载顺序。
4. 为未知物品、未知配方、未知附魔和未知设施定义保留策略，并让核心状态中的扩展内容统一使用 `(registryId, contentId)` 标识。
5. 在根存档增加按 `PackageId` 隔离的 `pluginData`，以不透明持久化信封保存任意插件私有状态；实现所有权检查、Schema 校验、逐版本迁移、单包与总量配额以及原子写入。
6. 打开存档前将其内容环境与当前目录的全局模组集比较，按兼容矩阵选择正常加载、复制迁移或拒绝打开，不得自动切换全局模组配置。
7. 提供一次性安全模式，只加载官方内容包、不改写全局启用状态，并禁止覆盖原存档。
8. 只有挂载器成功生成冻结的 `RegistrySnapshot` 后才创建 `GameApp`、Pinia Store 和游戏路由。
9. 实现跨版本存档导入：原版旧档使用官方迁移链，模组存档仅在环境一致或全部第三方迁移声明可用时允许在副本上尝试。
10. 实现通用 `packageSettings`：首版开放安装级配置并把内容配置摘要写入 lockfile 和存档环境；为后续存档级配置预留稳定容器，但不允许其改变注册表快照。
11. 在挂载器展示缓存命中、完整挂载、回退原因和耗时，并提供只清除派生缓存、不改变模组与存档的维护操作。

#### 验收标准

- 关闭含存档内容的数据包时，玩家会在写回存档前得到明确警告。
- 未知物品保留 ID、数量、品质和来源数据；重新启用数据包后可以恢复。
- 插件缺失或禁用时，其 `pluginData` 信封字段和 `payloadJson` 的 UTF-8 内容保持不变，且负载不被核心解释、迁移或改写；重新启用兼容插件后可以恢复。
- 插件只能读写自己的私有存档区；超出配额、Schema 校验失败或迁移失败时拒绝候选写入并保留旧数据，不得静默截断或清空。
- 安装级配置变化会重新构建候选快照并更新环境摘要；存档级配置不能改变包选择、加载顺序和注册表内容。
- 清除、损坏或缺少缓存只会触发完整挂载，不改变 lockfile、全局启用状态、配置和任何存档。
- 模组状态变化不会静默覆盖原存档，必要时创建备份槽或要求另存。
- 不同存档共享同一全局模组集；存档中的内容环境只用于诊断和保护，不参与模组启停决策。
- 原版旧档可以复制升级到新版目录；原版降级、模组环境不匹配和缺失迁移路径不会获得可写入口。
- 跨版本迁移失败不会修改旧版本目录中的原存档，新版成功结果只写入新版 `userdata/`。
- 桌面端实际启动未压缩版，验证数据包目录、存档目录和错误日志位置。

### 阶段 9：稳定化与首个公开数据包 API

#### 目标

冻结第一版可供第三方使用的内容 API，并提供示例和诊断工具。

#### 任务

1. 提供 `generate:schemas` 命令，从 TypeBox 定义生成并发布 JSON Schema、字段说明、ID 规则和兼容承诺；生成文件属于只读产物，不接受手工修改。
2. 提供最小示例数据包，并按全量迁移清单提供覆盖所有公开注册表类型的综合示例包或分域示例包集合。
3. 提供 `validate:schemas` 和独立数据包校验命令，在启动游戏前编译全部 Schema 并检查数据包。
4. 建立 `engineApiVersion` 与 `contentSchemaVersion` 的升级策略。
5. 发布 SemVer 支持范围、单包单版本规则、依赖类型、冲突诊断格式、稳定排序规则和 `mod-lock.json` Schema。
6. 发布配置作用域、本地化、作者与许可、资源署名、原子生命周期和 API 弃用规范，以及机器可读弃用索引。
7. 发布原版逐版本存档迁移链和兼容矩阵，明确模组存档跨版本不在官方保证范围内。
8. 建立 `valid/`、`invalid/`、`legacy/` 一致性测试包集合，让发布 CI 和第三方校验命令复用相同夹具及预期错误码。
9. 对加载性能、生命周期崩溃恢复、错误回滚和大数据包执行压力测试。
10. 发布持久化快照 Schema、缓存格式升级策略、环境键规则和官方黄金测试向量。

#### 验收标准

- 第三方作者无需修改项目源码即可制作和验证数据包。
- 公开 JSON Schema 可以从 TypeBox 源定义可重复生成，生成后工作区无差异；CI 在生成结果与已提交文件不一致时失败。
- 文档中的示例包可以在三端支持范围内成功导入。
- 兼容性承诺、弃用周期和错误码有正式文档。
- 每项稳定强制规则都有规则编号、机器校验、稳定错误码和一致性夹具；配置、本地化、发布元数据及生命周期规范均可由第三方工具验证。
- 模组作者在自己的 CI 中可以运行官方兼容测试入口，得到与游戏挂载器一致的结果。
- 冷挂载和缓存恢复有公开性能基线，旧缓存格式能明确失效或按已发布迁移规则升级。
- 原版升级、原版降级拒绝、模组环境不匹配和跨目录副本迁移都有自动化回归测试。
