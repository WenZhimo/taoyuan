# 重构协同与前置门槛

> [返回统筹说明](./README.md)

## 与现有重构计划的关系

项目现有 [重构计划](../refactor-plan.md) 已完成一轮大型 Store、View 和批处理职责收窄。模组系统不是再次全面重构的理由，也不应等待所有大文件拆完才开始。两者通过明确门槛协作：只做模组阶段所需的边界，不顺带重排无关 UI 和玩法。

## 阶段 0 前置门槛

开始注册表实现前必须满足：

- `pnpm run type-check`、`pnpm run lint`、`pnpm run validate:data`、`pnpm run validate:docs` 和完整测试有可记录基线。
- 旧存档夹具覆盖背包、配方、装备附魔、矿洞状态、NPC/任务进度和大型设施列表。
- 当前官方内容有规范化快照工具或测试夹具。
- [机器盘点](./content-inventory.json) 覆盖所有 `src/data/*.ts` 文件。
- 规范化哈希和错误码的首版规则已冻结。

## 阶段 1 前置门槛

开始迁移内容前必须建立以下最小代码边界：

```text
业务调用方
        → 内容查询门面
        → 旧静态适配器或 RegistrySnapshot
```

- View 和普通 composable 不新增对 `src/data/` 内容数组的直接导入。
- `getItemById()`、`getRecipeById()` 等现有入口先保留签名，内部来源可切换。
- 结构校验和语义校验有独立模块及测试。
- `RegistrySnapshot` 注入方式不会让 Pinia Store 在模块加载时捕获未初始化全局状态。
- 内容查询热路径使用索引，不因适配器引入重复线性扫描。

## 存档门槛

任何持久化 ID、`SaveContentEnvironment`、`packageSettings` 或 `pluginData` 改动开始前必须：

- 更新 `../save-compatibility.md`。
- 建立版本化旧存档夹具和失败回滚测试。
- 明确原版旧档、模组档、缺失包和降级场景。
- 所有迁移只在副本或候选状态上运行。
- 导入、保存、导出的大存档性能有基线。

## 允许并行的工作

| 工作流 | 可以并行的条件 |
| --- | --- |
| Schema 与 Registry 基础 | 使用固定夹具，不同时改玩法数值 |
| 官方内容快照与盘点 | 只读现有数据，不提前改调用方 |
| 挂载器 UI 原型 | 使用模拟诊断，不依赖未稳定内部类型 |
| 三端文件来源适配器 | 依赖稳定 `ContentPackageSource` 接口 |
| 本地化迁移 | `LocalizedTextRef` 和回退规则已冻结 |
| 兼容测试包 | 对应规则编号和错误码已登记 |

## 禁止并行的工作

- 同一内容域不得同时进行玩法数值调整和数据来源迁移。
- 存档根结构迁移不得与导入/导出性能重写混在同一提交。
- 注册表身份和命名空间规则未冻结前，不得批量重写持久化 ID。
- 官方内容尚未完全通过注册表运行前，不得开放第三方包绕过适配层。
- 生命周期事务未通过故障注入测试前，不得提供真实“升级”或“卸载”按钮。
- 事件和机制插件不得与第一阶段数据包并行实现。

## 阶段交付节奏

每个阶段拆成以下顺序：

```text
决策和 Schema
        → 失败夹具
        → 最小实现
        → 官方内容小切片
        → 等价性和性能验证
        → 文档与机器盘点更新
```

一次提交只承担一个逻辑职责。阶段未通过验收时保留旧适配路径，不删除旧数组、旧查询函数、旧存档读取或回归测试。

## 进入外载阶段的总门槛

阶段 7 之前必须证明：

- 官方内容包使用与第三方相同的清单、Schema、注册表和引用解析。
- 业务层不再直接依赖被迁移的静态内容数组。
- 官方内容快照和至少两套旧存档在三端逻辑层通过。
- 100,000 条定义的注册、校验和引用扫描持续提供进度且界面可响应。
- 无效官方包和第三方包都能完整回滚，不污染全局快照。
- Electron 构建会保留 `userdata/` 和 `mods/`，并能实际启动烟测。
