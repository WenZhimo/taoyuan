## 7. 内置数据剥离范围

### 7.1 应进入官方内容包

- 物品基础定义及来源文本。
- 标签定义，以及物品声明的多值标签。
- 作物、种子、果树、鱼类和采集物定义。
- 普通怪物、BOSS 和有序怪物池配置；怪物定义与池成员关系使用独立注册表。
- 标准附魔定义、稀有度和标准效果参数。
- 命名掉落表及掉落条目。
- 烹饪、加工和其他标准配方。
- 商店定义、商品、价格、库存及显示分组。
- 与上述内容直接关联的图标、图片和音频资源引用。

### 7.2 应留在游戏框架

- 战斗回合、伤害、暴击、状态效果和击败结算算法。
- 掉落概率超过 100% 时的倍率计算。
- 矿洞和骷髅矿穴的楼层生成算法。
- 商店营业时间判断、购买上限和交易动作。
- 种植、收获、烹饪和加工的状态变更流程。
- 存档序列化、迁移、未知内容占位和恢复策略。
- Vue 页面、弹窗、路由和分页组件。

### 7.3 需要拆分的混合文件

迁移不以文件为单位，而以职责为单位。例如：

- `items.ts`：定义进入内容包，自动生成逻辑进入内容编译器，查找函数改为访问注册表。
- `monsters.ts` 与 `mine.ts`：前者作为纯怪物定义和旧池映射的叶子数据源，怪物与池投影进入内容包；后者保留旧导出适配器、楼层生成、缩放和随机算法。
- `weapons.ts`：附魔定义已拆到 `enchantmentDefinitions.ts` 并投影为 `taoyuan:enchantment`，武器定义已拆到 `weaponDefinitions.ts` 并投影为 `taoyuan:equipment` 的 `kind: "weapon"` 分支；附魔效果汇总、价格计算、随机附魔、掉落结算和装备实例仍留在 domain/store 兼容路径。
- `processing.ts`：加工机器制造定义已拆到 `processingMachineDefinitions.ts` 并投影为 `taoyuan:processing_machine`，加工配方定义已拆到 `processingRecipeDefinitions.ts` 并投影为 `taoyuan:processing_recipe`；机器运行队列、多任务酒坊/种子制造机、日更结算和存档槽位仍留在现有 Store/框架路径。
- `shops.ts` 与 `ShopView.vue`：商店和商品定义进入内容包，营业判断和购买流程保留。

### 7.4 阶段 5 怪物迁移结果

- 21 个去重怪物定义进入 `taoyuan:monster`，掉落通过 `dropTableId` 引用 `taoyuan:drop_table`。
- 六个主矿洞区域池、六个 BOSS 楼层池和三个骷髅矿穴池进入 `taoyuan:monster_pool`。
- `MONSTERS`、`BOSS_MONSTERS`、`SKULL_CAVERN_MONSTERS` 和 `ZONE_MONSTERS` 继续由 `src/data/mine.ts` 原名重导出，作为兼容和回滚路径。
- `getFloor()`、弱化 BOSS、骷髅矿穴生成、矿洞 Store、矿洞页面和行会图鉴通过统一查询门面读取怪物池。
- 120 层区域映射、候选顺序、70% 弱化规则、深度阈值、25 层 BOSS 周期、缩放公式和随机调用位置仍属于框架且保持不变。

### 7.5 阶段 6 树木首切片结果

- 8 种果树和 3 种野树进入独立 `taoyuan:tree`，树苗、果实和采脂产物通过语义校验引用 `taoyuan:item`。
- `src/data/treeDefinitions.ts` 是旧格式树木数值的唯一静态叶子来源；`items.ts` 和官方适配器可以读取它，避免形成 `contentAccess -> staticAdapters -> fruitTrees/wildTrees -> contentAccess` 循环。
- `FRUIT_TREE_DEFS`、`WILD_TREE_DEFS` 继续由原模块原名重导出；`getFruitTreeDef()`、`getWildTreeDef()` 保留签名并在内部读取注册表兼容对象。
- 农场 Store、`useTreeActions.ts` 和 `ShopView.vue` 已通过统一门面读取树种、生长周期、季节产物、采脂周期和树苗信息。
- 种植、成长、换季、采脂器状态、收获、树木实例 ID 和存档字段仍属于框架且保持不变。

### 7.6 阶段 6 加工机器制造定义切片结果

- 20 种旧格式加工机器制造定义进入 `taoyuan:processing_machine`，制造材料通过语义校验引用 `taoyuan:item`。
- `src/data/processingMachineDefinitions.ts` 是旧格式加工机器制造数值的唯一静态叶子来源；`processing.ts` 继续原名重导出 `PROCESSING_MACHINES` 并保留 `getMachineById()` 兼容门面。
- `getMachineById()` 和 `getProcessingMachines()` 优先读取官方注册表并返回本地 ID 兼容对象；加工 Store、加工页面和派生机器物品仍可通过旧路径回滚。
- `src/data/processingCraftDefinitions.ts` 仅为洒水器、肥料、鱼饵、浮漂和炸弹建立无注册表依赖的叶子来源，用于打断内容适配器与 `items.ts` 的循环；这些内容域尚未在本切片迁移。
- 加工配方定义已由后续 `taoyuan:processing_recipe` 切片单独迁移；制造消耗顺序、退款、自动收取、酒坊/种子制造机多任务、加工队列、配方解锁和旧存档结构仍属于框架且保持不变。

### 7.7 阶段 6 加工配方定义切片结果

- 133 条旧格式加工配方定义进入 `taoyuan:processing_recipe`，机器引用通过语义校验指向 `taoyuan:processing_machine`，输入/输出物品通过语义校验指向 `taoyuan:item`。
- `src/data/processingRecipeDefinitions.ts` 是旧格式加工配方数值的唯一静态叶子来源；`processing.ts` 继续原名重导出 `PROCESSING_RECIPES` 并保留 `getProcessingRecipeById()` / `getRecipesForMachine()` 兼容门面。
- `getProcessingRecipeById()` 和 `getRecipesForMachine()` 优先读取官方注册表并返回本地 ID 兼容对象；加工 Store、加工页面、数据校验和旧配方静态导出仍可通过原路径回滚。
- 无输入配方以 `inputItemId: null` 和 `inputQuantity: 0` 表达；有输入配方要求 `inputItemId` 为内容 ID 且 `inputQuantity` 为正整数。
- 输入扣料、品质继承、自动收取、蜂箱/蚯蚓箱无输入自动重启、虚空箱回退、酒坊/种子制造机多任务、配方解锁和旧存档结构仍属于框架且保持不变。

### 7.8 阶段 6 秘密纸条定义切片结果

- 28 条旧格式秘密纸条定义进入 `taoyuan:secret_note`，宝藏奖励物品通过语义校验引用 `taoyuan:item`。
- `src/data/secretNotes.ts` 继续作为旧格式秘密纸条的唯一静态叶子来源，并保留 `SECRET_NOTES` 原名导出作为回滚路径。
- 注册表内部使用 `taoyuan:secret_note/{noteId}` 内容 ID；兼容门面继续向 Store 和 UI 返回旧 `SecretNoteDef` 数字 `id` 形状。
- `useSecretNoteStore.ts` 的总数、未收集随机池和使用奖励读取，以及 `AchievementView.vue` 的秘密纸条列表展示已改为通过官方注册表兼容门面读取。
- 钓鱼/挖矿触发收集的时机、随机调用次数、奖励发放顺序、`collectedNotes` / `usedNotes` 旧存档数字 ID 和存档 JSON 结构仍属于框架且保持不变。

### 7.9 阶段 6 教程提示定义切片结果

- 18 条旧格式晨间教程提示定义进入 `taoyuan:tutorial`，条件键限制为当前引擎已经支持的封闭枚举，不引入脚本式条件。
- `src/data/tutorials.ts` 继续作为旧格式教程提示的唯一静态叶子来源，并保留 `MORNING_TIPS` 原名导出作为回滚路径。
- 注册表内部使用 `taoyuan:tutorial/{tipId}` 内容 ID；兼容门面继续向隔夜结算返回旧 `MorningTipDef` 的本地 `id`、优先级、条件键和消息文本形状。
- `useEndDay.ts` 的晨间教程提示选择已改为通过官方注册表兼容门面读取；教程开关、已展示提示 ID、访问面板标记、临时标记清理和日志仍由既有框架处理。
- 旧存档 `tutorial.shownTipIds` 继续保存本地 `tipId` 字符串，存档 JSON 结构、条件判断、优先级排序和提示文案保持不变。
