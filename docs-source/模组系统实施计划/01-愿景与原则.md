# 愿景、范围与核心原则

> [返回统筹说明](./README.md)

## 1. 执行摘要

本计划的目标是把《桃源乡》从“游戏代码直接导入静态内容”演进为“游戏框架加载内容包”。改造完成后，当前内置物品、怪物、附魔、掉落、配方、商店商品、NPC、任务、成就、动物、作物、加工、装备、事件及其他静态内容都将组成官方内容包；官方内容包与第三方数据包通过同一套注册表、校验器和加载流程进入游戏。

本计划不以一次性重写为目标。迁移期间必须保持现有玩法数值、旧存档、Web、Electron 和 Android 构建可用。每个阶段只迁移一个边界明确的内容域，保留兼容适配层，并通过等价性测试证明迁移前后行为一致。

总体实施顺序如下：

```text
建立内容契约与注册表
        ↓
抽取首个官方内容包切片
        ↓
逐类迁移全部内置数据
        ↓
让官方内容完全通过注册表运行
        ↓
支持外部数据包发现、校验和加载
        ↓
提供模组管理、诊断和存档保护
        ↓
后续扩展事件插件与机制插件
```

## 2. 背景与当前状态

项目已具备较清晰的 Vue、Pinia、domain、composable 和组件边界，具体约定见 [架构与模块边界](../architecture.md)。这为后续扩展提供了基础，但当前内容数据仍属于编译期静态模块：

- 物品由 `src/data/items.ts` 中的数组汇总，并通过线性查找获得定义。
- 配方由 `src/data/recipes.ts` 直接导出。
- 怪物、区域怪物池、BOSS 和部分掉落共同位于 `src/data/mine.ts`。
- 附魔定义、附魔权重、效果映射和武器定义共同位于 `src/data/weapons.ts`。
- 商店定义位于 `src/data/shops.ts`，但不少商品列表和商店分支仍由 `ShopView.vue` 直接组织。
- `src/data/` 中还混有派生数据生成、随机算法、营业判断和流程辅助函数，并非全部都属于“内容”。

因此，第一步不能是把 `src/data/` 整体复制成 JSON，而是先建立内容边界和运行时注册表，再逐项剥离。

## 3. 目标与非目标

### 3.1 目标

1. 将 `src/data/` 中全部内置内容整理为版本化的官方内容包，包括 NPC、任务、成就等首批七类注册表之外的内容域。
2. 让官方内容和第三方数据包使用同一注册接口、校验规则和引用解析流程。
3. 支持新增物品、物品标签、怪物、附魔、掉落表、配方和商店商品，包括以标签匹配任意合格材料的配方。
4. 支持数据包依赖、加载顺序、命名空间、冲突诊断和版本兼容检查。
5. 保证缺少数据包时旧存档不会静默丢失未知内容。
6. 为 Web、Electron 和 Android 提供统一的逻辑加载接口，以及符合各平台限制的文件来源适配器。
7. 为后续事件插件和机制插件建立稳定的内容标识、生命周期和能力边界。
8. 将应用启动分为挂载器和游戏本体：挂载器先完成全局模组发现、信任校验、挂载和注册表冻结，再启动游戏本体。

### 3.2 非目标

1. 本阶段不支持任意 JavaScript 注入。
2. 本阶段不支持外部 Vue 单文件组件或运行时编译 `.vue` 文件。
3. 本阶段不支持游戏运行过程中热卸载数据包；启停数据包后需要重新启动或返回主菜单重新初始化运行时。
4. 本阶段不改变现有玩法数值，也不借迁移机会重做矿洞、商店、战斗或配方系统。
5. 本阶段不承诺任意旧版本数据包永久兼容，只承诺通过明确的内容 API 和迁移策略管理兼容范围。
6. 本阶段不保证模组存档在不同游戏版本、数据包版本或模组组合之间互通；官方兼容保证仅覆盖原版旧存档向原版新版本升级。

## 4. 核心设计原则

### 4.1 官方内容也走公开接口

官方内容包不得拥有绕过注册表的私有加载路径。只有当官方内容与第三方内容接受相同校验和解析时，扩展接口才是真实可用的。

### 4.2 数据、派生和算法分离

| 类型 | 示例 | 归属 |
| --- | --- | --- |
| 内容定义 | 青菜名称、售价、怪物生命、配方材料 | 内容包 |
| 派生内容 | 根据作物生成种子物品和收获物品 | 内容编译器 |
| 游戏算法 | 掉落倍率、伤害计算、矿洞楼层生成 | `domain/` 或 Store |
| UI 编排 | 商店弹窗、背包筛选、错误提示 | component、view、composable |
| 持久化状态 | 背包数量、已解锁配方、模组状态 | Store 和存档迁移 |

### 4.3 ID 是持久化 API

所有新内容使用 `数据包ID:本地ID` 格式，例如：

```text
taoyuan:cabbage
taoyuan:mud_worm
example_mod:purple_cabbage
```

现有旧 ID 在迁移期继续可读，并通过别名表映射到官方命名空间。显示名称可以修改，进入存档的 ID 不得随意修改。存档兼容要求见 [存档兼容与迁移约定](../save-compatibility.md)。

### 4.4 启动期可变，运行期冻结

注册表只允许在启动加载阶段注册内容。完成引用解析和校验后，注册表冻结为只读快照。这样可以避免运行过程中内容集合变化导致 Store、缓存、页面列表和存档出现不一致。

### 4.5 失败应可诊断、可回退

每个数据包先加载到包级暂存注册表，不得边解析边修改全局注册表。挂载器在暂存区完成格式修复尝试、Schema、依赖、引用和资源校验；只有整个候选环境通过后，才一次性生成冻结的 `RegistrySnapshot`。

自动修复采用严格白名单，只允许确定、无损、有版本规则且可重复执行的内存标准化：

- 按已发布迁移规则把旧 `contentSchemaVersion` 转为当前结构。
- 通过正式别名表转换旧字段名和旧内容 ID。
- 为缺失的可选字段补充 Schema 明确规定的默认值。
- 清除 UTF-8 BOM、规范化包内路径分隔符和其他不改变内容语义的表示差异。

禁止通过删除无效条目、忽略未知引用、虚构必填字段、替换缺失资源、修改概率/价格/数量或静默截断数据来“修复”数据包。签名包的原始文件不得被挂载器改写；修复只作用于暂存区中的规范化表示，原始签名仍针对原始文件清单进行验证。

每次修复都必须生成结构化 `RepairReport`，记录包 ID、文件、字段路径、修复规则 ID、修复前摘要和修复后摘要，并在挂载器中明确提示。修复后的整个候选环境仍必须重新通过完整校验，不能因为执行过修复而降低校验标准。

某个包在修复后仍无法通过校验时，必须丢弃该包的全部暂存条目和本次候选快照，强制终止启动流程，不得带着部分注册内容进入游戏本体。挂载器应显示数据包、文件、字段路径、错误码、已尝试修复和失败原因，并将完整诊断写入持久化错误日志。玩家可以修复/移除数据包后重试，或主动选择只加载官方内容包的安全模式；系统不得自动进入游戏并覆盖原存档。

官方内容包无法加载时，应回退到随程序发布且签名有效的只读副本；回退版本仍不通过校验时同样终止启动。
