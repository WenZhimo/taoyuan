# 内置内容迁移盘点

## 当前基线

截至 2026-07-14，`src/data/` 包含 45 个 TypeScript 文件，约 30,844 行和 427 个导出符号。对应机器可读清单见 [content-inventory.json](./content-inventory.json)。该清单已经逐文件记录职责分类、候选目标注册表和迁移阶段，但混合文件在实际迁移前仍必须完成导出符号级审计。

盘点不是“把整个文件变成 JSON”。每个导出符号必须归入以下类别之一：

| 类别 | 处理方式 |
| --- | --- |
| `content` | 进入官方内容包和公开注册表 |
| `derived` | 由内容编译器从基础定义稳定生成 |
| `algorithm` | 移入或保留在 `engine/domain` |
| `adapter` | 迁移期保留原函数签名，内部改查注册表 |
| `ui` | 留在主题、组件或玩家设置体系 |
| `barrel` | 仅负责重导出，不形成内容注册表 |

## 符号级审计要求

阶段 0 必须把机器清单扩展到每个公开导出符号，至少记录：

```ts
interface ContentInventorySymbol {
  file: string
  exportName: string
  classification: 'content' | 'derived' | 'algorithm' | 'adapter' | 'ui' | 'barrel'
  targetRegistry?: RegistryTypeId
  persistentIds: boolean
  consumers: string[]
  migrationPhase: number[]
  snapshotFixture?: string
  status: 'inventoried' | 'baselined' | 'migrated' | 'verified' | 'framework-retained'
  rationale?: string
}
```

机器盘点可以在顶层使用可选的 `reviewedArtifacts` 数组登记 `src/data/` 之外、但与当前迁移切片直接相关的查询门面或适配器符号。条目复用上述 `file`、`exportName`、`classification`、`targetRegistry`、`persistentIds`、`migrationPhase`、`status` 和 `rationale` 字段；`summary.reviewedArtifactCount` 必须与实际条目数一致。该数组不计入 `sourceRoot` 文件数或 `symbolCount`，避免混淆静态内容基线与迁移期代码产物。

不得只依据变量名分类。包含 `Math.random()`、日期、Store 访问或状态修改的函数默认按算法处理；数组和对象也可能是框架常量，需要根据消费者和存档语义判断。

## 阶段 5 审计结果

- `src/data/monsters.ts` 是怪物数值和旧池映射的唯一静态内容源，状态为 `verified`；注册表适配器不再依赖 `src/data/mine.ts`。
- `src/data/mine.ts` 中的 `MONSTERS`、`BOSS_MONSTERS`、`SKULL_CAVERN_MONSTERS` 和 `ZONE_MONSTERS` 是原名重导出的兼容适配器，不形成第二份可独立修改的数据。
- `ZONE_MONSTERS` 已投影为六个 `taoyuan:monster_pool` 区域池；BOSS 楼层映射和骷髅矿穴候选投影为另外九个池。
- `MINE_FLOORS`、`getFloor()` 的楼层规则、感染层、随机选择、弱化和缩放继续标记为 `framework-retained` 或适配器，不误归类为内容。
- `reviewedArtifacts` 已登记首切片的三个怪物查询门面，以及 `MonsterPoolDefSchema`、官方池适配器和五组怪物池查询门面，共 10 个阶段 5 审计产物。

## 阶段 6 树木首切片审计结果

- `src/data/treeDefinitions.ts` 新增为 8 种果树、3 种野树旧格式数值的唯一静态叶子来源，两个定义数组均标记为 `verified` 并指向 `taoyuan:tree`。
- `FRUIT_TREE_DEFS`、`WILD_TREE_DEFS` 标记为兼容重导出，`getFruitTreeDef()`、`getWildTreeDef()` 标记为注册表兼容查询；四个旧符号保留原模块路径与名称。
- `items.ts` 只为编译树苗和果实物品读取叶子定义，`staticAdapters.ts` 只为官方注册表投影读取，等价测试只用于迁移前后比较；这些路径不属于业务层绕过。
- `useFarmStore.ts`、`useTreeActions.ts` 和 `ShopView.vue` 已移除旧树木数组直连，新增符合现有模型的官方树种时无需修改这三个消费者。
- `reviewedArtifacts` 新增 `TreeDefSchema`、官方树木适配器和三组查询门面，共 5 个阶段 6 审计产物；清单当前覆盖 46 个 `src/data` 文件、30,844 行、429 个导出符号和 15 个审计产物。
- 鱼类、采集、动物、鱼塘、育种、生产、叙事和世界活动仍保持原盘点状态，未因树木切片误标为完成。

## 迁移顺序

1. `crops.ts`、`items.ts`、`recipes.ts`：建立第一条 Tag/Item/Crop/Recipe 闭环。
2. `shops.ts` 及各域商店池：统一 Shop 和 ShopOffer。
3. `weapons.ts`、`rings.ts`、`hats.ts`、`shoes.ts`：附魔、装备和掉落表。
4. `mine.ts`、`forage.ts`：怪物、区域池和战利品。
5. `processing.ts`、`animals.ts`、`fishPond.ts`、`breeding.ts`：生产和设施。
6. NPC、任务、事件、成就、博物馆、行会、教程和世界活动。
7. `timeConstants.ts`、`market.ts`、`hanhai.ts` 等高混合文件最后处理，先建立纯算法边界。

## 完成标准

“官方内容包完全接管”必须同时满足：

- 机器盘点中的每个文件和导出符号都有最终状态。
- 所有 `content` 和 `derived` 项已进入注册表或内容编译器。
- 所有 `algorithm` 和 `ui` 项有明确保留位置与理由。
- 所有持久化 ID 有别名或迁移策略。
- 全仓不存在业务层绕过查询门面直接导入旧内容数组的路径。
- 迁移前后规范化快照、固定随机测试和旧存档回归通过。
- 机器清单、文档、Schema 和官方包在 CI 中交叉校验。
