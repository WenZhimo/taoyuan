# 错误码规范

> [返回统筹说明](./README.md)

## 诊断结构

所有挂载、迁移、配置和存档错误使用统一结构。错误码稳定，显示文本可以本地化：

```ts
interface ModDiagnostic {
  code: string
  ruleId: string
  severity: 'info' | 'warning' | 'error' | 'fatal'
  stage: string
  messageKey: string
  packageId?: PackageId
  file?: string
  fieldPath?: string
  registryId?: RegistryTypeId
  contentId?: ContentId
  relatedPackageIds?: PackageId[]
  details?: Record<string, JsonValue>
  recovery: 'none' | 'retry' | 'disable-package' | 'remove-package' | 'safe-mode' | 'restore-backup'
}
```

日志可以包含技术堆栈，但面向玩家的诊断不得泄露私钥、完整本地绝对路径或其他插件的私有负载。

## 命名规则

错误码使用 `领域-类别-三位序号`：

```text
PKG-MANIFEST-001
PKG-DEPENDENCY-001
REG-REFERENCE-001
SAVE-PLUGIN-DATA-001
```

- 已发布错误码不得改作其他含义。
- 规则修正后保留旧码，必要时新增更具体的码。
- `messageKey` 可以变化和本地化，第三方工具只能依赖 `code`、`severity` 和结构化字段。
- 一个根因只产生一个主错误，其依赖链和字段问题作为结构化关联诊断，避免重复刷屏。

## 严重级别

| 级别 | 行为 |
| --- | --- |
| `info` | 状态说明，不影响候选环境 |
| `warning` | 可以继续，但必须显示并写入日志 |
| `error` | 当前包或操作失败，候选环境不得提交 |
| `fatal` | 安全、官方包或全局一致性失败，立即终止本次挂载 |

任一启用包产生 `error` 或 `fatal` 时，`GameApp` 不得创建。挂载器可以继续运行并允许玩家修复、禁用、移除或进入显式安全模式。

## 维护流程

新增错误码时必须同时提交：规则说明、诊断字段、玩家恢复动作、自动化测试和至少一个非法夹具。错误码登记表后续可以迁移为机器可读 JSON；在此之前本文件是编号事实来源。
