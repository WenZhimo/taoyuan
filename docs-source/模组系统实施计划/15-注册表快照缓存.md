> [返回统筹说明](./README.md)

> 配套规范：[确定性哈希](./10-规范化与哈希规范.md) · [错误码](./11-错误码规范.md) · [资源限制](./12-资源限制默认值.md)

## 基本结论

挂载器每次启动都必须运行，因为进程退出后内存注册表、索引和能力实例已经消失。但环境未变化时，不需要重新解析、迁移、校验和编译全部模组。系统应持久化纯数据化的编译注册表快照，下次启动验证环境后直接恢复。

```text
启动挂载器
        → 扫描包和全局配置
        → 计算当前 environmentHash
        → 命中并验证持久化快照
              ├─ 成功：恢复注册表、重建运行时索引、冻结快照
              └─ 失败：执行完整挂载并生成新缓存
        → 创建 GameApp
```

这是“每次逻辑挂载、按环境选择缓存恢复或完整挂载”，不是绕过挂载器，也不是复用已经退出进程的内存对象。

## 持久化模型

当前已经实现的是不依赖平台存储的 `SerializableRegistrySnapshot` v2 正文 codec。它保存可验证、可版本化的纯数据，并由 `SerializableRegistrySnapshotSchema` / `registry-snapshot.schema.json` 定义：

```ts
interface SerializableRegistrySnapshotV2 {
  formatVersion: 2
  registries: Array<{
    registryId: RegistryTypeId
    schemaName: string
    entries: Array<{
      owner: PackageId
      source: { packageId: PackageId; file?: string; localId?: string }
      entry: JsonObject & { id: ContentId }
    }>
  }>
  snapshotHash: string
}
```

注册表数组按 `registryIds()` 的稳定顺序保存；每个 `entries` 数组必须保留完整构建的真实注册顺序，不按 `contentId` 排序。数组位置本身就是顺序，不另存 index/order；这样 `snapshotHash`、恢复注册顺序和查询可见顺序共享同一事实来源。来源包 ID 和包内相对路径保留用于诊断和所有权检查；本地绝对路径、加载耗时、缓存命中时间等非确定数据不得进入正文。

未来磁盘缓存应在该正文外增加包含 `environmentHash`、`schemaSetHash`、内容编译器版本和签名信息的版本化 envelope；本切片没有实现该 envelope、目录/IndexedDB、签名或缓存命中逻辑。

v1 在生成时按内容 ID 排序，无法恢复原始注册顺序。它属于可删除、可重建的派生缓存，不是玩家存档；恢复入口明确拒绝 v1，未来缓存层收到 `CACHE-INVALID-001` 后应回退完整构建，而不是尝试迁移或静默接受错误顺序。

## 不得缓存的运行时对象

以下内容不能直接序列化进注册表快照：

- 函数、闭包、类原型和动态模块实例。
- Vue 响应式代理、Pinia Store 和组件状态。
- DOM、图片、音频、视频和字体解码句柄。
- Worker、文件描述符、IPC 对象和插件沙箱进程。
- 随机数生成器的可变运行状态。
- 绝对文件路径、临时 URL 和平台私有资源句柄。
- 未通过当前 Schema 和语义校验的原始外部 JSON。

资源注册表只缓存规范化资源描述和包内相对路径，实际媒体继续按需加载。未来事件或机制插件可以缓存其静态注册表定义，但插件代码、签名、能力授权和沙箱初始化仍须每次启动执行。

## 恢复流程

缓存命中后按以下顺序恢复：

```text
读取缓存头
        → 校验 cacheFormatVersion 和 environmentHash
        → 校验文件大小及 snapshotHash
        → 按当前持久化快照 Schema 解析
        → 恢复 RegistrySet 条目
        → 重建 Map、分类索引和只读包装
        → 校验条目计数及恢复后的快照哈希
        → freezeDefinitions() / freezeEntries()
```

当前 v2 codec 已实现 TypeBox 结构、版本、哈希、注册表集合/顺序、`schemaName`、条目 ID、owner/source 和 JSON 纯数据检查，并按数组顺序注册、冻结。未来缓存层可以不重新运行完整内容 Ajv 校验和跨注册表语义扫描，但必须执行缓存正文校验、条目计数、哈希和运行时不变量检查；任一步失败立即删除或隔离该缓存并转为完整挂载，缓存损坏不能阻止玩家通过完整挂载启动。

运行时 `Map` 和索引默认从缓存条目重建。只有当性能测试证明重建索引仍是瓶颈时，才增加版本化的持久化索引；索引缓存必须可从注册表正文重新生成，不能成为唯一事实来源。

## 缓存键与失效

`environmentHash` 至少覆盖：

```text
gameVersion
engineApiVersion
contentSchemaVersion
loaderVersion
contentCompilerVersion
schemaSetHash
cacheFormatVersion
信任策略和密钥撤销列表版本
每个启用包的 PackageId、version、contentHash、configurationHash
最终稳定加载顺序和依赖解析结果
```

以下任一变化必须导致缓存未命中：

- 新增、删除、启用、禁用或替换模组。
- 包版本、文件清单、`contentHash` 或安装级 `configurationHash` 变化。
- 依赖解析结果或稳定加载顺序变化。
- 游戏、引擎 API、内容 Schema、Schema 集、迁移器或内容编译器版本变化。
- 缓存格式、信任策略、发布者密钥状态或撤销列表变化。
- 缓存文件缺失、超限、结构非法或哈希不匹配。

文件大小和修改时间只能作为快速扫描提示，不能单独证明内容未变化。正式缓存命中必须绑定经过验证的文件清单和 `contentHash`。对于未来可执行插件，每次启动仍要重新验证代码签名和能力授权。

## 存储位置

| 平台 | 建议位置 |
| --- | --- |
| Electron | 程序目录 `userdata/mod-cache/<environmentHash>/` |
| Web | IndexedDB 的版本化 `mod-cache` object store |
| Android | 应用私有缓存目录中的版本化子目录 |

Electron 初始格式建议使用受 TypeBox Schema 约束的规范化 JSON；性能数据证明 JSON 解析成为瓶颈后，可以升级为 CBOR 或 MessagePack 等显式版本化格式。不得把 `v8.serialize()` 作为长期格式，因为 Node/Electron 版本变化可能破坏兼容性。

缓存属于可删除的派生数据，不进入存档、不随存档导出，也不参与玩家进度。清除缓存后只会触发下一次完整挂载。存档不得引用某个缓存文件才能读取。

## 原子写入和崩溃恢复

完整挂载成功后：

```text
生成 PersistedRegistrySnapshot
        → 写入临时缓存目录
        → 重新读取并验证结构和 snapshotHash
        → 原子重命名为 environmentHash 正式目录
        → 更新 lastSuccessfulMountHash
```

写入或验证失败时删除临时缓存，不影响当前已经验证的内存快照，也不覆盖上一次成功缓存。启动时清理过期临时目录，但不得删除仍被 lockfile 引用的最后成功缓存。

## 官方预编译快照

官方 `taoyuan-core` 可以随发布产物携带预编译注册表快照，使首次启动也无需重新编译全部官方数据。该快照必须：

- 在构建 CI 中从官方内容包确定性生成。
- 与官方包版本、`contentHash`、Schema 集和内容编译器版本绑定。
- 由官方发布签名覆盖。
- 在 Web、Electron 和 Android 构建中产生相同 `snapshotHash`。
- 验证失败时退回从官方内容包完整构建，而不是直接终止。

第三方包不能提供一个要求宿主无条件信任的最终全局快照。第三方可以附带自身预编译中间产物，但仍必须由挂载器验证来源、格式和环境键，首版不开放该优化。

## 可观测性

挂载器区分并记录：

```text
cache-hit
cache-miss-not-found
cache-miss-environment-changed
cache-miss-format-changed
cache-invalid-hash
cache-invalid-structure
full-mount
official-precompiled-hit
```

诊断记录恢复耗时、重建索引耗时、完整挂载耗时、条目数和缓存大小。不得记录插件私有数据内容或玩家存档内容。

## 测试要求

- 相同环境连续启动时第二次命中缓存，恢复后快照哈希与首次完整挂载一致。
- 任一包文件、配置、版本、加载顺序、Schema 或编译器变化都会失效。
- 截断、篡改、超限和旧格式缓存自动回退完整挂载。
- 缓存写入中断不会覆盖最后成功缓存。
- 冷挂载与缓存恢复的查询结果、排序、来源信息和派生内容完全一致。
- 对同一内容仅改变注册顺序会改变 `snapshotHash`；v2 往返恢复后顺序和哈希完全相同，v1 明确失效并触发完整重建。
- 100,000 条定义的缓存恢复显著快于完整校验，并保持挂载器进度反馈。
- 官方预编译快照在三端产生相同 `snapshotHash`。
- 清除缓存不改变 lockfile、模组启用状态和任何存档。
