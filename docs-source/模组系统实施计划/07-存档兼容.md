## 10. 存档兼容方案

### 10.1 全局模组集

模组启用状态属于当前程序目录/安装实例的全局配置，不属于某个存档，也不跨不同版本目录共享。挂载器启动时先读取 `GlobalPackageConfiguration`，完成发现、信任确认、依赖检查和注册表挂载，再启动游戏本体。当前目录中的所有存档共享当前冻结的注册表快照。

```ts
interface GlobalPackageConfiguration {
  enabledPackageIds: PackageId[]
  trustedPublisherFingerprints: string[]
  lastSuccessfulMountHash?: string
}
```

存档不得要求挂载器自动启用或下载其中记录的数据包。打开存档时只比较当前全局模组集与保存时的内容环境，并按后续兼容矩阵决定正常加载、复制迁移或拒绝打开。

### 10.2 内容环境元数据

建议在存档中增加：

```ts
interface SaveContentEnvironment {
  saveFormatVersion: number
  gameVersion: string
  engineApiVersion: string
  contentSchemaVersion: string
  packages: Array<{
    id: PackageId
    version: string
    contentHash: string
    configurationHash: string
    loadIndex: number
  }>
}
```

`contentHash` 必须参与比较，防止数据包作者在不修改版本号时替换内容；`configurationHash` 对内容相关安装级配置的规范化结果求哈希，没有此类配置时使用规定的空配置哈希；`loadIndex` 保存最终稳定拓扑顺序。程序目录的 `userdata/mod-lock.json` 是当前全局环境的可重现解析结果，存档中的 `SaveContentEnvironment` 是保存时的历史快照，二者必须分别维护。模组系统发布前产生且没有这些字段的历史存档可以按原版旧存档处理；模组系统发布后的存档缺少必要环境字段则视为环境信息损坏，不得猜测其模组组成。

现有没有统一版本号的官方历史存档在通过旧形状识别和完整性校验后标记为 `saveFormatVersion: 0`。此后每次格式变化都必须提供 `vN → vN+1` 的确定迁移，不允许跳过中间步骤；无法识别来源形状或任一步失败时保留原文件并终止迁移。

原版存档精确定义为：仅启用当前游戏随附并签名有效的 `taoyuan-core`，没有任何第三方数据包、事件插件或机制插件。

### 10.3 版本兼容矩阵

官方兼容承诺采用以下边界：

| 存档环境 | 当前环境 | 策略 |
| --- | --- | --- |
| 原版旧版本 | 原版新版本 | 官方保证按逐版本迁移链升级 |
| 原版新版本 | 原版旧版本 | 禁止降级读取和写入 |
| 模组旧版本 | 相同模组的新版本 | 由模组作者决定，官方不保证成功 |
| 模组环境 A | 不同模组环境 B | 默认不兼容，不允许直接覆盖原档 |
| 模组存档 | 原版环境 | 不正常加载；只允许诊断、保留未知内容或在副本上尝试 |
| 原版存档 | 模组环境 | 先创建副本，再允许进入模组环境 |
| 游戏版本、包版本和内容哈希完全一致 | 相同环境 | 正常读取和写入 |

官方只保证原版低版本存档向原版高版本游戏升级，不保证不同模组组合、不同模组版本或不同游戏版本的模组存档互通。第三方模组可以自愿提供迁移声明，挂载器负责验证和执行，但迁移结果不属于官方兼容保证。

### 10.4 跨版本导入与迁移

跨版本迁移必须以导入副本为入口：

```text
在新版目录选择旧存档
        ↓
复制到新版临时区并保留原文件
        ↓
读取 SaveContentEnvironment
        ↓
原版旧档：执行官方逐版本迁移
模组旧档：检查精确环境或全部第三方迁移声明
        ↓
在暂存区完成迁移、注册表和存档校验
        ↓
成功后另存到新版 userdata/
```

任何失败都必须丢弃新版临时结果并保留旧目录原文件。新版不得把迁移结果写回旧版本目录，也不得自动把新格式存档同步给旧版游戏。模组环境不一致且缺少完整迁移路径时，默认拒绝可写加载；诊断或安全模式不得覆盖原档。

降级默认不提供逆向迁移。玩家需要回到旧版本目录继续使用旧存档，或恢复升级前备份；新版保存过的存档不能交给旧版直接打开。

### 10.5 裸 ID 兼容

现有存档中的 `cabbage` 等 ID 不应立即重写并使旧版本无法读取。推荐策略：

1. 注册表查询先识别完整命名空间 ID。
2. 对旧裸 ID 查询官方别名表。
3. 新存档逐步写入标准 ID，但在过渡发布周期内保持双读。
4. 导出存档前不强制全量重写未知字段。

### 10.6 缺失内容

缺失数据包时不得直接过滤未知 ID。最低保留要求：

- 物品：ID、数量、品质、锁定状态和组成标签；旧存档缺少组成标签时补为空数组。
- 武器和附魔：原始定义 ID、附魔 ID 列表和实例状态。
- 配方：已解锁 ID。
- 设施和副本状态：原始命名空间数据块。

UI 使用“未知内容（原 ID）”占位展示，并禁止会破坏数据的操作。重新加载对应数据包后恢复正常定义。

### 10.7 插件私有存档区

已知内容的未知占位不能覆盖未来任意自定义注册表和新机制的运行时状态。根存档必须提供统一、按 `PackageId` 隔离的 `pluginData`，避免每增加一种机制都修改核心存档结构。

插件运行时看到的逻辑契约为：

```ts
interface PluginDataEnvelope {
  schemaVersion: string
  payload: unknown
}

type PluginData = Record<PackageId, PluginDataEnvelope>
```

为保证插件缺失时核心无需解析即可原样保留，实际持久化使用不透明 JSON 字符串信封：

```ts
interface PersistedPluginDataEnvelope {
  schemaVersion: string
  encoding: 'json'
  payloadJson: string
  payloadHash: string
}

interface PluginDataContainer {
  pluginData: Record<PackageId, PersistedPluginDataEnvelope>
}
```

`payloadJson` 只允许保存 JSON 兼容数据，不允许可执行代码、函数、原型对象或外部文件句柄。`payloadHash` 使用 `payloadJson` UTF-8 字节的 SHA-256，校验不透明负载在读取、复制和重新保存过程中没有被意外修改。根存档 Schema 只验证信封，不验证未知插件的负载结构。

插件存在且版本兼容时，由插件提供其私有数据各版本的 Schema 和连续迁移链：

```text
读取本插件的不透明信封
        → 校验 payloadHash
        → 解析 payloadJson 为 unknown
        → 按旧 schemaVersion 校验
        → 依次执行 vN → vN+1 严格迁移
        → 按当前 Schema 校验
        → 通过引擎受控 API 交给插件
```

迁移只作用于候选副本。任一步失败时保留原始信封并拒绝可写加载，不得猜测字段、删除未知数据或执行逆向迁移。插件的 `schemaVersion` 独立于游戏版本、引擎 API、内容 Schema 和插件包版本；插件升级不一定需要改变私有数据版本。

插件缺失、禁用或当前版本无法识别该 `schemaVersion` 时，核心必须把持久化信封作为不透明数据保留：信封字段值不变，`payloadJson` 字符串及其 UTF-8 内容和哈希不变。核心不解释、不迁移、不规范化负载，也不因重新保存其他游戏状态而删除或改写它。这里保证的是不透明负载内容不变，不要求根存档文件的缩进和字段排列逐字节相同。重新安装兼容插件后再恢复解析。该保留规则本身不授予存档可写权限；是否允许打开仍由内容环境兼容矩阵决定。

每个插件默认只能通过受控存储 API 读取和替换自身 `PackageId` 下的数据，不能枚举、读取或修改其他插件的私有区。跨插件共享状态必须使用另行审核的公开注册表、事件或显式能力，不能直接访问对方 `pluginData`。更改 `PackageId` 默认视为新插件；所有权迁移必须由受信任的显式迁移声明完成。

引擎必须公布单包容量、全部插件数据总容量、JSON 深度、节点数量和单次增长限制。容量按未压缩 `payloadJson` 的 UTF-8 字节数计算，不能依靠压缩绕过。插件提交更新时按“序列化候选负载 → Schema 校验 → 配额校验 → 哈希计算 → 与游戏状态原子保存”的顺序执行；任何失败都保留旧信封。导入的历史信封超限时应保留原文件并拒绝可写加载，给出包 ID、实际大小和限制值，不得静默截断、清空或覆盖。

自定义注册表与 `pluginData` 职责不同：注册表保存随插件发布、启动时可重建的静态定义；`pluginData` 保存某个存档中的可变实例状态。新船型、新技能定义属于注册表，玩家拥有的船和已解锁技能属于 `pluginData`。两者不得相互替代。

### 10.8 数据包设置持久化

配置定义随包发布，配置值属于用户数据。安装级配置按 `PackageId` 保存在平台全局配置目录；存档级配置使用根存档唯一的通用容器，不为每个机制新增字段：

```ts
interface PersistedPackageSettings {
  schemaVersion: string
  values: Record<NamespacedId, JsonValue>
}

interface SaveData extends PluginDataContainer {
  packageSettings: Record<PackageId, PersistedPackageSettings>
}
```

引擎根据包提供的设置 Schema 校验值、补充新增默认项并执行连续迁移。玩家已经显式设置的值不得因包升级或默认值变化而被覆盖。包缺失时保留该包的整个设置区域但不展示可编辑表单；重新安装兼容版本后恢复。删除废弃设置必须通过显式迁移，不能仅因新 Schema 不再声明该键而自动丢弃。

安装级配置在构建候选 `RegistrySnapshot` 前解析，任何变化都触发重新挂载，并把规范化配置哈希写入 lockfile 和 `SaveContentEnvironment`。存档级配置只能在存档通过环境检查后注入游戏运行时，不得影响依赖解析、内容注册或快照哈希。配置迁移和保存使用候选副本并与对应的全局配置或存档事务原子提交；失败时保留旧配置和旧存档。
