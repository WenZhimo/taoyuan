# 核心玩法规则约定

本文记录容易被重复实现或误改的核心规则。修改这些规则时，必须同步 domain、调用方、功能测试和性能测试。

## 掉落率倍率

权威实现：`src/domain/drops/rollChanceQuantity.ts`。

掉落概率参数不是只能位于 0 到 1 的百分比，而是期望掉落次数：

```text
掉落数量 = floor(max(0, chance)) + 对小数部分再判定一次
```

示例：

| chance | 结果 |
| --- | --- |
| `-0.5` | 固定 0 |
| `0.75` | 75% 概率掉 1 个 |
| `1` | 固定掉 1 个 |
| `1.5` | 固定掉 1 个，另有 50% 概率再掉 1 个 |
| `2` | 固定掉 2 个 |
| `2.7` | 固定掉 2 个，另有 70% 概率再掉 1 个 |

约束：

- 不要把概率强制截断到 100%。
- 所有支持倍率溢出的掉落都应复用 `rollChanceQuantity()`。
- 测试随机分支时注入固定随机函数，不要依赖真实 `Math.random()`。
- 倍率计算顺序属于玩法规则，调用方应先合并加成，再交给统一函数取整。

## 时间表示

- 一天从 `6` 点开始。
- 午夜表示为 `24`。
- 凌晨 1 点表示为 `25`。
- 凌晨 2 点表示为 `26`，即 `PASSOUT_HOUR`。
- `advanceTime()` 达到或超过 `26` 时将时间固定为 `26` 并返回晕厥结果。

不要把跨午夜时间转换回 `0` 至 `2`，否则会破坏排序、晚睡恢复和小憩上限计算。

## 小憩

权威规则：

- `src/domain/sleep/napRules.ts`
- `src/composables/layout/useSleepFlow.ts`
- `src/composables/layout/useRestActions.ts`

规则：

1. 玩家可以在任意地点打开小憩。
2. 输入分钟数向下取整，非有限值回退为 1，最少 1 分钟。
3. 小憩不能到达凌晨 2 点；最大时长为“距离 2 点的分钟数减 1”。
4. 实际推进时间调用 `gameStore.advanceTime(actualMinutes / 60, { ignoreSpeedBuff: true })`。
5. 小憩不受食物速度增益影响。
6. 每小时恢复最大体力的 12%，按实际经过分钟数折算并向下取整。
7. 只要实际经过时间大于 0 且仍缺体力，至少恢复 1 点；恢复量不超过缺失体力。
8. 请求时间超过可用时间，或推进时间返回晕厥时，显示“小憩被打断”。
9. 体力恢复必须使用实际经过时间，而不是玩家请求时间。

## 睡袋

睡袋 ID 为 `sleeping_bag`。

拥有判定：

- 背包或临时背包中携带睡袋；或
- 任意仓库箱子中存在数量大于 0 的睡袋。

可原地过夜的地点组：

| 地点组 | 醒来页面 |
| --- | --- |
| `nature` | `forage` |
| `mine` | `mining` |
| `hanhai` | `hanhai` |

地点判定优先使用当前路由对应地点组，无法识别时再使用 `gameStore.currentLocationGroup`。

行为：

- 有睡袋时，正常睡觉或凌晨 2 点晕厥都可在资源地点过夜并醒在原地点组。
- 矿洞睡袋过夜会清理战斗状态，但保留矿洞探索数据。
- 没有睡袋时使用普通隔夜流程；矿洞探索会结束并返回常规醒来地点。
- 睡袋不消耗数量。
- 睡袋只改变过夜地点与副本保留方式，不绕过晚睡或晕厥恢复规则。

## 物品堆叠

权威实现：

- `src/domain/inventory/itemStacks.ts`
- `src/stores/useInventoryStore.ts`

规则：

- 单栈最大数量为 `999_999_999`。
- 只有 `itemId` 和 `quality` 都相同的物品才合并。
- 添加顺序为：已有主背包栈、主背包空槽、已有临时背包栈、临时背包空槽。
- domain 堆叠函数通过 `remaining` 返回未放入数量；Store 会保留已放入部分、提示剩余物品丢失，并返回 `false`。
- 购买或批量发放前可用 `canAddItem()` 检查能否完整放入，避免先扣钱后溢出。
- 未指定品质移除物品时，消耗顺序为普通、良品、优品、极品。
- 整理背包会合并同 ID 同品质物品，并按最大栈重新拆分，保留锁定状态。
- 当前反序列化会过滤未知物品并克隆有效物品，但不会自动拆分超大栈；未来若改变最大栈或接收外部存档，应在迁移层显式规范化数量并补测试。

“足够大”不等于 JavaScript 无穷大。数量必须保持有限整数，避免 `Infinity`、精度丢失和无法序列化。

## 农田和温室批量上限

权威实现：`src/domain/farm/batchLimits.ts`。

- 农田和温室的内部批次大小均为 1000 项。
- 玩家只需确认一次，系统会自动连续处理全部可操作目标。
- 超过 1000 项时显示实时进度、已处理数量和取消按钮。
- 每个内部批次完成后让出主线程，使界面动画和取消操作能够响应。
- 取消操作时保留已经完成的批次，并退还尚未使用的预扣种子、肥料或体力。
- 1000 项只限制单个内部批次，不限制一次玩家操作或农田、温室的总地块数量。

新增批量动作时必须：

1. 复用 `shouldConfirmLargeBatch()` 和 `runChunkedBatch()`，不要要求玩家重复点击。
2. 先取得稳定快照，避免处理中数组变化导致漏项或重复。
3. 不在每个条目中重复执行全量统计或生成逐项日志。
4. 预扣资源的批量动作必须在取消或提前停止时退还未使用部分。
5. 为 10 万级数据准备自动完成、取消边界和性能测试。

## 列表分页

权威实现：`src/composables/game/usePagination.ts`。

- 默认每页 50 项。
- 页码始终限制在 `1..totalPages`。
- 空列表的 `totalPages` 仍为 1，避免 UI 出现第 0 页。
- 数据量减少导致当前页越界时，使用 `safePage` 自动回到有效页。
- 分页只解决渲染数量；统计和日结算热路径仍需避免重复全量扫描。

育种台、酒窖、牲口棚、鱼塘等可能无限增长的列表应复用该 composable，而不是各自维护页码算法。

## 特殊物品与数据一致性

特殊物品的 ID、兑换数量和品质应集中在 `src/data/`，不要在多个 View 或 Store 中复制常量。

当前“墨墨的fumo”由 `src/data/specialItems.ts` 统一定义：

- 青菜 2000 个。
- 输出 fumo 2000 个。
- 品质固定为极品。
- 普通 NPC 视为最爱礼物。
- 隐藏 NPC 视为共鸣供奉。

新增或修改内容数据后必须运行 `pnpm run validate:data`。

## 修改规则的完成标准

- domain 规则测试覆盖正常值、边界值和异常输入。
- 调用方测试证明状态和副作用正确。
- 大数据量或高频调用有性能边界。
- 存档中的相关字段或 ID 保持兼容。
- 文档、数据校验、全量测试和构建全部通过。
