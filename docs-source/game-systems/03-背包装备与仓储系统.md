# 背包装备与仓储系统

> 本文档基于当前 `src/` 源码重新调研生成，并已补充状态字段说明与更完整的核心机制流程。`docs/` 是构建输出目录，请继续在 `docs-source/game-systems/` 维护手写文档。

## 系统定位

说明背包容量、物品堆叠、工具升级、装备、戒指、钱包能力和仓储。

## 核心机制

- 背包不是单纯 itemId 列表，而是带容量、数量、品质和特殊临时区的物品栈系统。奖励写入时必须检查容量；同一物品不同品质不能随意合并，否则会破坏售价、料理、任务提交或博物馆判断。满包时临时区和失败提示也需要参与流程。
- 工具、武器、戒指、帽子、鞋子和钱包能力都在库存体系附近维护。工具影响行动效率，武器影响矿洞战斗，戒指和钱包能力提供跨系统被动效果，例如作物成长、钓鱼稳定、商店折扣或采矿体力减免。
- 仓库是背包之外的长期容器，箱子有容量和等级，虚空箱这类特殊箱子会在 `GameLayout.vue` 提供全局存取入口。仓库存取不应绕过背包规则，仍要尊重数量、品质、容量和重复堆叠逻辑。
- 工具升级是延迟完成机制：发起升级时扣材料和钱，写入 `pendingUpgrade`，跨日倒计时结束后才替换工具等级。这个机制避免升级成为即时战力变化，也让跨日结算承担“完成项目”的统一职责。
- 装备与钱包能力属于被动效果源，真正生效点分布在农田、商店、矿洞、钓鱼和采集等消费方。新增装备效果时，不能只在背包显示解锁，还要在对应玩法逻辑中读取并应用。

## 关键约束

- 背包写入必须处理容量、品质和临时区，不能只按 itemId 判断成功。
- 装备/钱包效果是被动能力，新增效果必须在消费方读取，否则只会显示已解锁但没有实际影响。

## 跨系统联动

- 与时间系统联动：动作消耗、跨日推进、每日重置或过晚边界需要保持一致。
- 与背包/经济联动：奖励、材料扣除、购买出售和容量失败需要走统一 store。
- 与长期进度联动：新增动作应考虑技能经验、成就统计、任务进度、图鉴发现或博物馆捐赠。

## 维护清单

- 修改状态字段时，检查对应 `serialize` / `deserialize` 是否需要更新。
- 修改每日推进时，检查 `useEndDay.ts` 中的调用顺序和日志。
- 修改数据配置时，检查商店、图鉴、成就、任务和 UI 展示是否引用同一 itemId。
- 新增 UI 弹窗时，检查 `GameLayout.vue` 是否需要暂停实时计时或处理全局状态。
