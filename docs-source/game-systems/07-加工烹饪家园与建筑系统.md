# 加工烹饪家园与建筑系统

> 本文档基于当前 `src/` 源码重新调研生成，并已补充状态字段说明与更完整的核心机制流程。`docs/` 是构建输出目录，请继续在 `docs-source/game-systems/` 维护手写文档。

## 系统定位

说明加工机器、加工队列、烹饪食谱、buff、家园建筑、洞穴、地窖和升级。

## 核心机制

- 加工系统是“输入物 -> 机器 -> 等待天数 -> 输出物”的队列。启动时扣除材料并占用机器/队列，跨日 `dailyUpdate()` 才减少剩余天数并产出成品；这意味着小时级跳时不应直接完成加工。
- 烹饪是即时消耗食材的系统，但食谱解锁是长期网络：技能等级、NPC 好感、节日、成就、瀚海特殊物品都可能解锁新食谱。料理还可能提供 buff，影响速度、体力或其他玩法，所以食谱既是经济产物也是能力入口。
- 家园建筑提供永久能力：农舍、温室、洞穴、地窖、工坊等升级会改变容量、产物池、陈酿效率或可用功能。它们通常消耗材料和金钱，并在后续跨日流程中持续产生影响。
- 地窖和洞穴都是“选择/投入后等待”的长期结算点：洞穴按类型和等级每日产物，地窖按槽位和周期提升品质或价值。它们不像商店那样即时交易，而是让玩家把资源转成未来收益。
- 加工、烹饪和建筑共享材料消耗问题。启动前必须检查背包、仓库或资金是否足够；启动后要保证失败不扣材料，成功则由统一 store 写入队列或解锁状态。

## 关键约束

- 加工和地窖按天推进，不能用小时跳过直接完成。
- 食谱解锁来源分散，新增食谱时要检查技能、NPC、节日、成就和瀚海触发路径。

- 与时间系统联动：动作消耗、跨日推进、每日重置或过晚边界需要保持一致。
- 与背包/经济联动：奖励、材料扣除、购买出售和容量失败需要走统一 store。
- 与长期进度联动：新增动作应考虑技能经验、成就统计、任务进度、图鉴发现或博物馆捐赠。

## 维护清单

- 修改状态字段时，检查对应 `serialize` / `deserialize` 是否需要更新。
- 修改每日推进时，检查 `useEndDay.ts` 中的调用顺序和日志。
- 修改数据配置时，检查商店、图鉴、成就、任务和 UI 展示是否引用同一 itemId。
- 新增 UI 弹窗时，检查 `GameLayout.vue` 是否需要暂停实时计时或处理全局状态。
