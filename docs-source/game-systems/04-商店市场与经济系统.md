# 商店市场与经济系统

> 本文档基于当前 `src/` 源码重新调研生成，并已补充状态字段说明与更完整的核心机制流程。`docs/` 是构建输出目录，请继续在 `docs-source/game-systems/` 维护手写文档。

## 系统定位

说明购买、出售、市场价格、旅行商人、金币流转和商店库存。

## 核心机制

- 经济系统的核心不只是商店列表，而是“价格来源 + 资金流 + 库存/日期条件”。`usePlayerStore.ts` 的 money 是资金事实来源，商店购买、出售、任务奖励、瀚海贸易和晕倒惩罚都应通过统一接口体现，避免出现多个余额来源。
- `useShopStore.ts` 根据季节、商店类型、解锁状态和折扣效果生成可购商品。种子、肥料、钓具、装备、药品、服饰等商品来自不同 data 文件，但最终都要落到扣钱与入包；若背包满或材料不足，交易必须失败并保持资金不变。
- 出售逻辑需要读取物品基础价格、品质、市场修正和可能的技能/钱包加成；销售历史还会影响统计、成就或出货记录。因此“把物品换成钱”同时也是成就、图鉴、经济平衡和任务进度事件。
- 旅行商人是日期驱动的库存：用年/季节/日期生成稳定随机种子，当天重复打开不会变，但换日或特定日程会刷新。这样既保留随机性，也避免玩家通过反复打开界面刷新商品。
- 市场与商店连接几乎所有产出系统：农田、钓鱼、矿洞、畜牧、加工和瀚海物品最终都可能进入出售或购买链路。新增商品时要确认 itemId、价格、店铺开放条件、折扣、库存刷新和展示分类。

## 关键约束

- 资金变化要汇入 `usePlayerStore.ts`，不要让商店或小游戏维护第二套余额。
- 价格、折扣、市场波动和旅行商人库存应可由日期/季节复现，避免打开界面反复刷新。

## 跨系统联动

- 与时间系统联动：动作消耗、跨日推进、每日重置或过晚边界需要保持一致。
- 与背包/经济联动：奖励、材料扣除、购买出售和容量失败需要走统一 store。
- 与长期进度联动：新增动作应考虑技能经验、成就统计、任务进度、图鉴发现或博物馆捐赠。

## 维护清单

- 修改状态字段时，检查对应 `serialize` / `deserialize` 是否需要更新。
- 修改每日推进时，检查 `useEndDay.ts` 中的调用顺序和日志。
- 修改数据配置时，检查商店、图鉴、成就、任务和 UI 展示是否引用同一 itemId。
- 新增 UI 弹窗时，检查 `GameLayout.vue` 是否需要暂停实时计时或处理全局状态。
