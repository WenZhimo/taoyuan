# 技能成就博物馆与收藏系统

> 本文档基于当前 `src/` 源码重新调研生成，并已补充状态字段说明与更完整的核心机制流程。`docs/` 是构建输出目录，请继续在 `docs-source/game-systems/` 维护手写文档。

## 系统定位

说明技能经验、成就统计、图鉴发现、博物馆捐赠、收集包和秘密纸条。

## 核心机制

- 技能系统是动作反馈层：农耕、钓鱼、采矿、采集、战斗、烹饪等动作发放经验，等级提升后影响体力消耗、品质、可制作/可烹饪内容或 perk 选择。技能不是独立小游戏，而是对主玩法熟练度的长期记录。
- 成就系统保存长期统计，而不是只在成就页计算。每次收获、钓鱼、赚钱、下矿、烹饪、击杀、育种都应在对应动作发生时记录，之后 `checkAchievements()` 才能可靠判断。若动作没有记录，成就页无法事后准确推断。
- 图鉴发现、秘密纸条、博物馆和收集包构成收藏网络。获得新物品要记录发现；可捐赠物要进入博物馆判断；收集包提交会扣材料并记录完成；秘密纸条可能解锁奖励或提示。
- 这套系统常常是“旁路消费者”：它不一定发起玩法动作，但必须监听或被调用来记录玩法结果。新增系统最容易漏掉的就是这里的统计与收藏联动。
- 长期进度也会反向解锁玩法：成就、技能等级、博物馆奖励或秘密纸条奖励可能提供食谱、能力、道具或目标，从而影响经济和后续行动。
- 从维护角度看，这些记录层应该跟动作源尽量靠近：收获时记录作物，钓鱼成功时记录鱼，怪物死亡时记录击杀，而不是等玩家打开成就页再扫描当前状态。这样可以保留“曾经发生过”的历史事实。

## 关键约束

- 新增玩法必须显式记录统计，否则成就页无法“事后猜出”玩家做过什么。
- 新物品要决定是否可发现、可捐赠、可提交收集包或可触发秘密纸条。

## 跨系统联动

- 与时间系统联动：动作消耗、跨日推进、每日重置或过晚边界需要保持一致。
- 与背包/经济联动：奖励、材料扣除、购买出售和容量失败需要走统一 store。
- 与长期进度联动：新增动作应考虑技能经验、成就统计、任务进度、图鉴发现或博物馆捐赠。

## 维护清单

- 修改状态字段时，检查对应 `serialize` / `deserialize` 是否需要更新。

- 修改数据配置时，检查商店、图鉴、成就、任务和 UI 展示是否引用同一 itemId。
