# 存档云同步重置与架构系统

> 本文档基于当前 `src/` 源码重新调研生成，并已补充状态字段说明与更完整的核心机制流程。`docs/` 是构建输出目录，请继续在 `docs-source/game-systems/` 维护手写文档。

## 系统定位

说明存档序列化、导入导出、云同步、重置、设置和长期维护边界。

## 核心机制

- 存档不是一个 store 的快照，而是由 `useSaveStore.ts` 汇总各业务 store 的 `serialize()` 结果，再写入槽位、导出文本或云同步。读取时再分发给各 store 的 `deserialize()`，因此每个业务 store 都要对自己的持久字段负责。
- 反序列化必须兼容旧档。新增字段时，`deserialize()` 应提供默认值；删除或重命名字段时，要考虑旧数据迁移，否则玩家读档后会出现空状态、undefined 行为或 UI 展示异常。
- 重置和新游戏流程需要同时清空业务状态、UI 状态和持久化槽位引用。仅清空一个 store 会留下跨系统残留，例如背包没清但农田重置、任务仍指向旧物品、NPC 好感仍保留等。
- 导入导出和云同步要处理版本、加密/解密、槽位选择和失败回滚。导入失败不能污染当前存档，云同步失败也不能让本地槽位进入半写入状态。
- 构建产物和手写资料分离也是架构约束：`docs/` 会被 Vite 清空，长期文档必须放在 `docs-source/`，源码分析产物放在 `docs-source-analysis/`。

## 关键约束

- 新增持久字段必须补 serialize/deserialize，并为旧档提供默认值。
- 手写文档继续放在 `docs-source/`，不要放回构建输出 `docs/`。

## 跨系统联动

- 与时间系统联动：动作消耗、跨日推进、每日重置或过晚边界需要保持一致。
- 与背包/经济联动：奖励、材料扣除、购买出售和容量失败需要走统一 store。
- 与长期进度联动：新增动作应考虑技能经验、成就统计、任务进度、图鉴发现或博物馆捐赠。

## 维护清单

- 修改状态字段时，检查对应 `serialize` / `deserialize` 是否需要更新。
- 修改每日推进时，检查 `useEndDay.ts` 中的调用顺序和日志。
- 修改数据配置时，检查商店、图鉴、成就、任务和 UI 展示是否引用同一 itemId。
- 新增 UI 弹窗时，检查 `GameLayout.vue` 是否需要暂停实时计时或处理全局状态。
