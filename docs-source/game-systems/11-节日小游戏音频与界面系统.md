# 节日小游戏音频与界面系统

> 本文档基于当前 `src/` 源码重新调研生成，并已补充状态字段说明与更完整的核心机制流程。`docs/` 是构建输出目录，请继续在 `docs-source/game-systems/` 维护手写文档。

## 系统定位

说明节日与小游戏弹窗、音频/BGM、教程、设置、主题和全局 UI 编排。

## 核心机制

- 节日和小游戏通过对话/弹窗进入，核心是“特殊日程 -> 弹出活动 -> 暂停或覆盖主循环 -> 奖励回写”。活动结束后要把奖励、BGM、日志和全局状态恢复到正常游戏，避免玩家卡在活动态。
- `useAudio.ts` 不是简单播放音乐，它会根据季节、天气、战斗、节日、小游戏和瀚海覆盖选择不同 BGM，并应用天气与时间段的音色/节奏修正。音频状态因此与游戏时间、天气和地点都有关。
- `GameLayout.vue` 是全局编排层：状态栏、睡觉/小憩、日志、虚空箱、宠物领养、儿童提议、农场事件、设置弹窗都在这里协调。新增全局弹窗时要确认它是否暂停实时钟，以及关闭后是否恢复原来的 BGM/输入状态。
- 教程和设置系统属于 UI 体验状态：它们不会直接改变经济循环，但会改变提示出现、字体、主题、消息样式和用户操作节奏。教程标记通常由玩法动作写入，再由提示系统读取。
- 小游戏组件应只处理局部交互和表现，最终奖励必须写回正式 store，例如背包、金钱、成就或节日奖励记录。否则刷新页面或读档后，小游戏成果无法被长期系统感知。

## 关键约束

- 全局弹窗需要明确是否暂停实时钟，活动结束要恢复 BGM 和普通游戏状态。
- 小游戏奖励要写回正式 store，而不是只停留在组件局部状态。

## 跨系统联动

- 与时间系统联动：动作消耗、跨日推进、每日重置或过晚边界需要保持一致。
- 与背包/经济联动：奖励、材料扣除、购买出售和容量失败需要走统一 store。
- 与长期进度联动：新增动作应考虑技能经验、成就统计、任务进度、图鉴发现或博物馆捐赠。

## 维护清单

- 修改状态字段时，检查对应 `serialize` / `deserialize` 是否需要更新。
- 修改每日推进时，检查 `useEndDay.ts` 中的调用顺序和日志。
- 修改数据配置时，检查商店、图鉴、成就、任务和 UI 展示是否引用同一 itemId。
- 新增 UI 弹窗时，检查 `GameLayout.vue` 是否需要暂停实时计时或处理全局状态。
