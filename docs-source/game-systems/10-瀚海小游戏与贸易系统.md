# 瀚海小游戏与贸易系统

> 本文档基于当前 `src/` 源码重新调研生成，并已补充状态字段说明与更完整的核心机制流程。`docs/` 是构建输出目录，请继续在 `docs-source/game-systems/` 维护手写文档。

## 系统定位

说明瀚海区域、贸易、轮换库存、小游戏、赌局、特殊材料和瀚海解锁。

## 核心机制

- 瀚海是一个独立区域，但仍复用主游戏经济。解锁后会出现贸易点、贸易商店等级、周轮换库存、每日下注次数和特殊材料；这些状态都保存在瀚海 store，而不是散落在小游戏组件里。
- 贸易系统通过特殊货物和轮换商品制造长期目标。购买/兑换时会扣钱、扣贸易点或扣材料，并把瀚海物品送入背包；这些物品又可解锁加工或烹饪内容，因此贸易不是孤立商店，而是资源输入源。
- 小游戏和赌局有每日限制。下注、倍率、胜负和奖励都在 `useHanhaiStore.ts` 中结算，跨日时 `resetDailyBets()` 清空次数，周边界刷新轮换库存。这样可以避免玩家通过反复进入组件刷新收益。
- 瀚海音乐、地点分组和旅行成本也接入主系统：进入瀚海不是单独页面切换，而是时间、地点、音频、商店和背包共同参与。
- 瀚海特殊材料会向外部系统扩散，例如香料、丝绸、仙人掌、枣等可进入食谱或加工配方。新增瀚海物品时，要同步物品主表、商店显示、加工/烹饪解锁和可能的成就统计。
- 因为瀚海同时包含贸易和小游戏，它的核心风险是收益刷新边界不清。所有“每日”与“每周”限制都应由 store 和跨日流程控制，组件只负责表现一次局部玩法结果。

## 关键约束

- 每日下注和每周库存必须在跨日/跨周逻辑中重置，不能只在界面打开时刷新。
- 瀚海特殊材料会向烹饪、加工和商店扩散，新增物品要全链路登记。

## 跨系统联动

- 与时间系统联动：动作消耗、跨日推进、每日重置或过晚边界需要保持一致。
- 与背包/经济联动：奖励、材料扣除、购买出售和容量失败需要走统一 store。
- 与长期进度联动：新增动作应考虑技能经验、成就统计、任务进度、图鉴发现或博物馆捐赠。

## 维护清单

- 修改状态字段时，检查对应 `serialize` / `deserialize` 是否需要更新。
- 修改每日推进时，检查 `useEndDay.ts` 中的调用顺序和日志。
- 修改数据配置时，检查商店、图鉴、成就、任务和 UI 展示是否引用同一 itemId。
- 新增 UI 弹窗时，检查 `GameLayout.vue` 是否需要暂停实时计时或处理全局状态。
